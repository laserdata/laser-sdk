# Examples and testing

Examples are runnable demos of the coordination substrate. The integration suite is the automated coverage. Both run against a real Apache Iggy instance. Load [laser-sdk-overview](../laser-sdk-overview/SKILL.md) first. Repo rules in [AGENTS.md](../../../AGENTS.md).

## STOP and ask the user before

- Making the SDK call an LLM. The SDK is LLM-agnostic transport. Model calls live in example/app code behind the `LlmClient` seam only.
- Wiring examples as CI tests, or changing the shared-container / one-stream-per- test isolation in `harness.rs` (it keeps the suite parallel and isolated).

## Examples (`examples/rust/`)

- One crate, `[[example]]` bins under `src/<scenario>/main.rs`, run with `cargo run --example <name>` against `just up`.
- **Shared connection: `laser_examples::connect()`** (in `src/lib.rs`). Every example builds its `Laser` via `laser_examples::laser(&stream_for("<example>"), caps)` over `connect()`, so connection is resolved in one place from env, local-or-cloud with no code change. **Stream isolation: `stream_for(example)`** returns `LASER_STREAM` if set, else a per-example `laser-<example>` stream. Each example gets its own stream so they can all run against one local server without colliding on the well-known agent topics (AGDX isolates workloads by stream, never by partition - spec B1.1). Never share one stream across unrelated example apps. Default is local Iggy (`iggy://iggy:iggy@127.0.0.1:8090`). Two cloud inputs: `LASER_CONNECTION_STRING` (a full iggy string with creds embedded - `iggy+tcp://user:pwd@host` or `iggy+tcp://<token>@host`. A bare `host` form is scheme-normalized), or `LASER_SERVER=<host>` + auth (`LASER_TOKEN`, or `LASER_USERNAME`+`LASER_PASSWORD`). `with_resolved_tls` only auto-attaches our CA for LaserData hosts (`is_laserdata_host` = `*.laserdata.cloud`), and does so even when `tls=true` was passed without a `tls_ca_file=` (it injects the CA), a string that already names `tls_ca_file=`, a non-LaserData host, or `LASER_NO_TLS` pass through. The CA is chosen by subdomain via `is_dev_host` - a `.sandbox`/`.dev` host uses `laserdata_dev.crt`, else `laserdata.crt` (both vendored at `examples/certs/` (examples-level, language-agnostic), embedded via `include_bytes!`). `host_of` strips scheme/userinfo/port, `attach_tls` does the param assembly. Overrides: `LASER_TLS_CERT`, `LASER_NO_TLS`. When adding an example, reuse `connect()` - never hardcode an endpoint. Pure helpers (`host_of`, `is_laserdata_host`, `is_dev_host`, `attach_tls`, `ensure_default_port`) are unit-tested in `lib.rs`.
- **Shared projector: `laser_examples::start_projector(laser, topic, content_type, fields)`**. Query is a managed (LaserData Cloud) feature, so this registers the projection on the Cloud (`register_cloud_projection` publishes `laser.projections().register(..)` plus `laser.bindings().apply(..)`) and lets the Cloud project. It returns a `Projector` handle whose `shutdown()` is a no-op. Against raw Apache Iggy the registration and the query both return `LaserError::Unsupported`, so the query examples need a LaserData Cloud deployment. Query topics use `_` not `.` (`api_calls`), since LaserData Cloud materializes each index under that exact name, `[A-Za-z0-9_]` only.
- `src/lib.rs` holds the **`LlmClient` seam**: `trait LlmClient { async fn complete(...) }` + `MockLlm` (deterministic, key-free, what CI/docs use) and `AnthropicLlm` (`llm-anthropic`, reads `ANTHROPIC_API_KEY`/`ANTHROPIC_MODEL`) and `OpenAiLlm` (`llm-openai`, reads `OPENAI_API_KEY`/`OPENAI_MODEL`), both via reqwest. `default_llm()` picks real-when-feature-and-key (Anthropic, then OpenAI), else MockLlm. Keep all model calls in examples, never in `sdk/`.
- What examples prove: the substrate, not model intelligence - provenance/ causality, reliable delivery + DLQ, dedup, context windows, fan-out, handoff, sessions, memory, `agdx.deadline`-bound tool calls. Keep them deterministic.
- Six examples, each a complete realistic system (not one call per binary):
  - `event-analytics` (generic): one clickstream topic, every read model over it. A live `iggy_consumer_group` ticker tails the raw log while the producer streams (offsets commit server-side via `AutoCommit::When(PollingMessages)`, the configuration that cannot starve on redelivery, with a timeout diagnostic instead of a hang), query aggregates (`count`/`group_by`, `order_desc`, reserved `message_type`/`ts` via `time_range`), a resumable `Cursor` + `StateStore` export, and cloud-gated schema codas: a JSON Schema guard (a malformed event never materializes) plus raw Avro datums decoded by a registered writer schema (sync register returns LaserData Cloud-allocated id, `.add_avro` publishes). Uses `start_projector`.
  - `order-book` (generic, coda cloud-gated): the latency-minded market profile. A tuned raw `iggy_producer` streams fills in paced bursts while a tight-poll `iggy_consumer_group` folds a live book, the fills index to a queryable tape body-first (integer cents stay typed), and the Avro coda registers a Fill writer schema (sync register returns LaserData Cloud-allocated id) and publishes raw datums via `.add_avro`, aggregating LaserData Cloud-decoded columns.
  - `firehose` (generic): the load generator. Millions of multi-KB messages across many org indexes (gigabytes) to drive LaserData Cloud's ingest + query path under storage pressure (`LASER_FIREHOSE_MESSAGES/ORGS/CONCURRENCY/ PAYLOAD_BYTES`).
  - `concierge` (agentic, partly cloud-gated): the AI support desk. A ticket firehose materializes the world model (records carry `message_type` + `ts`), semantic memory (`QueryMemory` + `Embedder`) recalls past resolutions, four agents coordinate only by topics: triage fans deadline-bounded `ctx.request` tool calls (`join_all`) and synthesizes via the `LlmClient` seam, a specialist answers from recall, a resolver with a KV `Deduplicator` (`Agent::builder().deduplicator(..)`) applies credits effectively once behind a durable `HumanInput` approval gate, and a speculative fork (`put_row`, overlay query, left open by default, `LASER_APPLY_PLAN=1` promotes or squashes) stages the bulk-resolution. A `coordination_demo` phase shows three coordination primitives on one connection: a credit-ledger compare-and-swap (`set(..).expect_absent()` / `.expect_version(v).commit()` with an `is_version_conflict()` retry loop), a `read_your_writes()` query, and `LaserError::code()` classifying any outcome into the unified `ResultCode` (so an unserved level reports rather than fails). The incident rebuilds from its conversation with `ConversationState::load`. Each managed phase skips gracefully without LaserData Cloud.
  - `interop` (agentic): edge interoperability over the log. One LLM-backed `Worker` (the `LlmClient` seam) sits behind both the `A2aBridge` (Commands/Responses) and the `McpBridge` (ToolCalls/ToolResults). The example drives an A2A `message/send` â†’ `tasks/get`, an MCP `tools/call`, an AG-UI render of a chat chunk stream via `agui_events`, and a human-in-the-loop pause (`Agdx::request_input` on `HumanInput`, resolved by an approver agent's `AgentCtx::respond_input`). Pure open-SDK (no cloud gating), runs on MockLlm or a real `llm-*` backend. Mirrors the integration tests in `a2a.rs`/`mcp.rs`/`agui.rs`/`human_input.rs`.
  - `memory` (agentic, partly cloud-gated): agentic memory, both halves over one incident-knowledge domain. Part 1 is the four memory verbs as one loop in process over `VectorMemory` with a deterministic bag-of-words `Embedder` (`remember`, semantic `recall`, feedback-weighted `improve`, `forget`), so it needs no server. Part 2 is the durable knowledge graph: `upsert` services and components as content-addressed nodes and typed edges (`GraphNode::entity` / `GraphEdge::relate`, `mitigated_by` edges stamped with a bitemporal `valid_from`), read a node's `neighbors`, traverse from a predicate (`start_match` + `out`), trace an incident's blast radius, read the graph `as_of` a past instant, and return whole `paths`. Each entity is also written as a record in a `topology` key-value namespace and its node/edge carries a `SourceRef` (`GraphEdge::with_source`) pointing at that record, so the source is a live deep link the console renders as a click-through; the example prints a node's source to show the trace-back-to-origin story. The graph half connects only when it runs, so the memory half stays serverless. It skips cleanly without LaserData Cloud (`graph` capability) and the built graph is browsable in the console explorer.
  - `orchestra` (agentic, open-SDK): one orchestrator over a pool of capability agents, all coordination over the log. Five phases: discovery (diagnose agents auto-advertise capability cards, resolved from the fused registry), a directed `Laser::contract` (`to_capable`, deadline, ack-on-pickup), a scatter `Laser::scatter` panel (every capable agent, fold the findings), health exclusion (an agent advertising `Unavailable` is skipped), and `Laser::quarantine` (an operator pulls an agent, the next panel routes around it). Fixed inbox route so it runs on a stock local Apache Iggy. Verified to run live end to end. The Python peer is `examples/python/orchestra.py` (the same five phases over the PyO3 `contract`/`scatter`/`quarantine` bindings).
- The cloud-gated phases connect with `Capabilities::OPEN`, check `managed_kv` / `managed_host` / `forks` (set by the capability handshake) via the `fork_feature_ready` guard, and without LaserData Cloud print "needs LaserData Cloud" and exit `Ok` - so they stay green in CI. The crate enables the SDK's `a2a-bridge` + `agui` + `kv` + `mcp-bridge` + `query` + `schema-codecs` features.

## Integration tests (`sdk/tests/integration/`)

- `main.rs` registers each scenario as a module. `harness.rs` owns `TestIggy` (shared `OnceCell` container) and `laser()` (a fresh bootstrapped `Laser` on a unique stream per test). `eventually(...)` polls instead of fixed sleeps.
- `iggy_container.rs` is the `TestIggy` testcontainer wrapper - test-only, lives under `tests/integration/`, not shipped in the crate. Pin the image via `LASER_TEST_IGGY_TAG` (default `edge`).
- Gated by the zero-dep `integration` feature so `cargo test --workspace` (default features) stays unit-only. Run with `cargo test --workspace --all-features` (or `just test-it`). Needs Docker. The full set enables `a2a-bridge` (the A2A integration test), `kv`, and `query`.
- The managed surface (query, KV, forks) needs LaserData Cloud and is not run end-to-end here. Query is the off-log `AGDX_QUERY` managed command, KV and forks are raw managed commands, all returning `Unsupported` on raw Apache Iggy. There is no in-process query worker and no topic query path. The split: the client wire bytes are pinned by the fixture corpus, the full managed execution is verified in LaserData Cloud's own repository (it consumes this published SDK), and the **query DSL semantics** are covered here by a pure in-memory reference engine (`laser_bdd::query_engine` (in the `bdd/` harness), no Iggy, no transport, no qr) through its own unit tests and the `bdd/scenarios/query.feature` scenarios. So every SDK has a reference for what a query must return. This repo otherwise tests the SDK against vanilla Apache Iggy only, so every test runs for any contributor with Docker.

## Cross-SDK BDD conformance (`bdd/`)

- The cross-language contract that every future SDK port must pass. `bdd/scenarios/*.feature` is one shared Gherkin set, and `bdd/rust/` is the cucumber-rs reference runner. Everything runs against a real Apache Iggy testcontainer with NO managed backend and NO Cloud: streaming, capability negotiation + the Unsupported boundary (now including compare-and-swap and a `read_your_writes` query, each asserting the unified `ResultCode` is `Unsupported`), provenance, the AGDX agent envelope (including a pure-wire must-understand scenario: an envelope `.requiring()` a feature bit a receiver lacks has an unmet requirement), and query (served by the SDK's in-process query worker from `laser-examples`, since query rides a log topic). Run with `just bdd` (or `cd bdd/rust && cargo test`). `LASER_BDD_ADDR=host:3000` points at an external server for the docker-compose multi-language path.
- KV and fork *behavior* is NOT BDD-tested here: they are raw Iggy command codes (`send_raw_with_response`) only the managed backend dispatches, so the success paths cannot run without it. What the BDD does cover for them is the Unsupported boundary on open Iggy (e.g. the compare-and-swap scenario) and the unified result-code classification. Their byte compatibility is covered by the fixture corpus, and their success behavior is tested in LaserData Cloud's own repository. We never run BDD against a Cloud deployment.
- `bdd/rust` is a crate OUTSIDE the workspace (its own `[workspace]`), so `cargo test --workspace` does not touch it and it has its own fmt/sort/machete/clippy gate (the `lint-detached` CI job, `just lint`). Same for the `fuzz/` crate.
- Two complementary conformance layers: the wire fixture corpus pins the bytes (every envelope, every surface), the BDD scenarios pin the behavior. A new SDK is conformant when it passes both. Step definitions reuse the SDK's public API and do not fork the scenarios per language. Details: `bdd/README.md`.

## Rules specific to this area

- Test names are BDD (`given_..._when_..._then_should_...`), assertions use `.expect("message")`, never bare `.unwrap()`.
- Add a new scenario as both an example (demo) and, where it guards behavior, an integration test. New behavior fixes should land an edge-case test (timeout, DLQ, shutdown, isolation), not only a happy-path one.
- To exercise a raw/edge wire case (e.g. a header-less message), publish via the Iggy producer directly (`iggy` is an sdk dev-dependency) rather than through `send_agent`, which always stamps provenance.

## Review smells

- An LLM call leaking into `sdk/` (must stay behind `LlmClient` in examples).
- A new fix with only a happy-path test and no edge case.
- Fixed `sleep`s in integration tests instead of `eventually`.
- A test that shares a stream with another test (breaks isolation).
