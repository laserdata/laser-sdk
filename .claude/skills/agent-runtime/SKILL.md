# Agent runtime

`agent/` is the customer-facing runtime: how you send, consume reliably, route, and spawn agents. Load [laser-sdk-overview](../laser-sdk-overview/SKILL.md) first. repo rules in [AGENTS.md](../../../AGENTS.md).

## STOP and ask the user before

- Changing public signatures: `Laser::{send_agent, request, bootstrap, spawn_subconversation}`, `AgentHandler::handle` (takes `&AgentMessage` + an `&AgentCtx<'_>`), `AgentCtx`, `Agent`/`AgentHandle`, `AgentConsumer::run`. These are the API customers code against.
- Changing the commit mode (`AutoCommit::After(ConsumingEachMessage)`) - it is what makes delivery at-least-once + commit-after-handle.
- Changing dedup-before-handle ordering or the DLQ behavior.

## Key files and symbols

- `laser.rs` - `Laser` facade.
  - `send_agent`: encode provenance -> headers, partition by `provenance.partition_key()`, send via a **per-topic cached producer**.
  - Producer cache: `DashMap<String, Arc<OnceCell<Arc<IggyProducer>>>>`. The shard lock is held only to fetch/insert the cell. `producer.init().await` runs inside the `OnceCell`, off the map. Never hold any map lock across the network init - that would serialize unrelated sends behind one slow topic.
  - `request`/`AgentReplyReader`: poll the reply topic with a **per-partition offset cursor** so each poll reads only new messages, not a full rescan from 0. Correlation is a fresh `idempotency_key` (Ulid) stamped on the request. The responder echoes it back via `AgentCtx::respond` and `next_match` filters on the key. `conversation_id` alone is NOT used for correlation, so a forged reply that guesses the conversation id cannot hijack the request. `partitions_count` is cached on first call so a 30s timeout does not do 300 `get_topic` round-trips.
  - `spawn_subconversation(&Provenance) -> Provenance`: fresh `conversation_id`, `parent_conversation_id` = parent's, `root_conversation_id` carried through.
  - `bootstrap`/`ensure_topic`/`ensure_stream`: idempotent create-if-missing. `bootstrap` also warms a producer per well-known topic concurrently.
- `consumer.rs` - `AgentConsumer::run` + `ReliableConsumer` (impls Iggy's `MessageConsumer`). Per message: decode (undecodable -> `dead_letter_undecodable`, `DeadLetterReason::DecodeFailed`, payload verbatim). **Target-agent filter** (skip if `provenance.target_agent_id` is set and does not match this consumer's `agent` id, defensive guard for the one-agent-one-group invariant documented on `Agent`). Dedup on `idempotency_key` via the `Deduplicator` seam, deadline check -> DLQ, then the handler with retry/backoff. The per-task fence high-water map is bounded: idle entries are swept past a ttl once it grows over a soft cap, so a long-lived consumer's map stays bounded by its active working set. A handler error that `is_retryable()` retries to `max_attempts` then DLQs. A `LaserError::Rejected` (from `LaserError::rejected(..)`) DLQs immediately, no retry. Every DLQ path publishes a typed `AgentDeadLetter` capsule (via `dead_letter`/`dead_letter_undecodable` -> `publish_dead_letter`): the `DeadLetterReason` (`DecodeFailed`/`DeadlineExceeded`/`Rejected`/`RetryExhausted`), the attempt count, the poison message's full `LogPosition` (built from `stream_id`/`topic_id` resolved once in `run`), and the original payload verbatim. The capsule message keeps the original provenance headers (deadline cleared, so a deadline-bound DLQ consumer does not re-drop it) and is marked `agdx.ct = cbor`. DLQ publication is best-effort: a failed publish is logged at `error!` with the reason and the offset still commits, so the loss is loud, not silent. `run` races `consume_messages` against a shutdown receiver, and fires a `ready` sender once the consumer has joined its group and is polling. Its poll interval defaults to a reactive 10ms (`POLL_INTERVAL`), overridable per agent.
- `agent/laser.rs` - `Laser::redrive_dead_letter(&AgentDeadLetter)`: reads the original record at the capsule's `source` log position (numeric stream/topic ids), then republishes it verbatim (headers + body via `user_headers_map`), re-keyed by conversation, to its source topic. Errors (`LaserError::Invalid`) if the source stream/topic is gone or the record aged out of retention.
- `AgentMessage::from_received` uses `message.header.offset` (the message's own offset), NOT `received.current_offset` (the partition high-water, shared across a polled batch). Gotcha worth keeping: the high-water would give every message in a batch the same offset.
- **Envelope-aware consume.** `from_received` detects an AGDX message via the `agdx.av` header (`is_agdx`), decodes the `AgentEnvelope`, and exposes it on `AgentMessage.envelope: Option<AgentEnvelope>` (a decode failure dead-letters as `DecodeFailed`). The runtime `provenance` is then *synthesized from the envelope* (`provenance_from_envelope`: conversation/source/target/idempotency/deadline), so the target filter, dedup, and deadline checks read one shape for AGDX and `send_agent` messages alike. Agent ids are name strings on both sides, so `source`/`target` map straight across (a name the SDK validator rejects just drops out). Non-AGDX messages keep `Provenance::try_from` and `envelope = None`. The decoded envelope carries the `must_understand` u64 bitset (A9.1): a receiver calls `envelope.unmet_requirements(understood)` (where `understood` is the OR of the `agent::features` bits it implements) and rejects/dead-letters a message with any unmet bit rather than mis-handling a feature it lacks. No feature bits are defined yet, so today every well-formed message has `must_understand == 0` and the check is a no-op awaiting the first strict-handling feature.
- `Provenance::try_from(&IggyMessage)` matches keys first and fails-not-skips: a known provenance key with a non-string value is a decode error, while foreign/unknown keys are ignored. The AGDX typed headers (`agdx.ct` u8, `agdx.av` u32, the `Uint128` routing duplicates) coexist on one record because they are foreign to the provenance dictionary, not because non-string values are skipped.
- `Deduplicator` seam (`async fn observe(key) -> bool`, `async_trait`, `dyn`-safe): default `SlidingWindow` (in-memory bounded set over the private `DedupWindow`). A durable backend drops in via the optional `deduplicator` on `Agent::builder` / `AgentConsumer` - back it with any `StateStore` (`get`/`set`/`delete`). The managed `Kv` implements `StateStore`, so `laser.kv(ns)` is the durable drop-in (see the concierge's resolver, which uses a KV dedup). Warm dedup pre-fills whichever impl is configured. Async + dyn so a premium backend can do I/O.
- `Laser::capabilities()` -> `Capabilities` (`capabilities.rs`): the premium set the connected streaming infrastructure advertises. Defaults to `OPEN` (all false) on raw Apache Iggy. A managed runtime injects the negotiated set via `Laser::with_capabilities(..)`. Reserve premium branches behind it. Never assume premium is present. The concierge's cloud-gated phases show the pattern (the `fork_feature_ready` capability gate).
- `builder.rs` - `Agent::builder` (id/listen_on/handler/optional respond_on/optional `poll_interval`/`inbox_route` defaulting to `InboxRoute::Advertised`/`capabilities`) -> `spawn(Laser) -> AgentHandle`. The `inbox_route` sticks as the agent's routing convention and flows builder -> consumer -> ctx for the directed-send and fan-out helpers. When `capabilities` is non-empty the spawned agent self-advertises before consuming (the free `advertise` fn): publishes a non-expiring capability card to the registry (`Laser::publish_card`) and, where the server serves it, advertises its `listen_on` topic as live presence (`advertise_presence`, query-gated, best-effort). Empty `capabilities` keeps the prior private-agent behavior. `ack_on_pickup` (off by default) makes the reliable consumer emit a `Working` task status on `respond_on` the instant an AGDX command is picked up, before the handler, so a `Laser::contract` caller can distinguish consumed from expired-unconsumed. `AgentHandle`: `ready()` waits until the agent has joined its group and is polling (await it before publishing so nothing races the join, idempotent), `shutdown()`/`join()` surface `LaserError`, `abort()`. **dropping it keeps the agent running** (detached), matching the old behavior - only an explicit `shutdown()` (an `Ok(())` on the channel) stops it.
- `ctx.rs` - `AgentCtx<'a>` handed to `handle`: `respond` (reply on `respond_on`, chained + routed to sender), `reply_on` (chained reply on an explicit topic), `send`, `request`, `respond_input` (resolve the human-in-the-loop interrupt being handled: a `response` carrying the handled command's interrupt correlation, the responder half of `Agdx::request_input`), `fan_out(selector, payload, GatherPolicy, deadline) -> Gather` (resolve every capable agent, route each branch to its own inbox via the ctx `InboxRoute`, gather replies on the orchestrator's `respond_on`; a target with no resolvable inbox is a surfaced branch failure, never a shared-topic fallback), `approval_gate(reply_topic, prompt, timeout) -> Vec<u8>` (pause the handler on a human decision: an interrupt on the human-input topic, awaiting the approver's reply, `Rejected` on rejection; a thin wrapper over `Agdx::request_input`), `spawn_subconversation`, `laser`/`message` accessors. It is how handlers act without storing `Laser`. The consumer derives `ctx.agent` from the group name and carries the builder's `respond_on` + `inbox_route`.
- `agdx.rs` - the typed AGDX producer: `Laser::agdx(topic, source, conversation) -> Agdx` with the verbs `command`/`respond`/`emit`/`status`/`fail` (each returning an `AgdxSend` builder that validates the envelope at `send()`), `request_input(reply_topic, prompt, timeout)` (human-in-the-loop interrupt/resume: a prompt `command` under a fresh minted interrupt correlation, then awaits the correlated `response` on `reply_topic` and returns its body, or `LaserError::Rejected` when the reply is an `error` terminal - composes existing verbs, no new wire), and `stream(correlation, purpose) -> AgdxStream` (auto-incremented `sequence`, the purpose + abandonment bound on the opening chunk, one terminal via `finish` or `fail`). Every send stamps `agdx.av` (u32), `agdx.ct` (u8), the conversation as a typed `Uint128` header and the target as the agent's name string, and partitions by the conversation's canonical base32 form. `AgdxStream::write` rejects bodies above `MAX_CHUNK_BODY_BYTES` before publish and increments `sequence` only after the append succeeds. The draft chunking guidance consts (`DEFAULT_CHUNK_FLUSH_BYTES`, `DEFAULT_CHUNK_LINGER_MS`, `MAX_CHUNK_BODY_BYTES`) live here.
- `assembler.rs` - `ChunkAssembler`, the pure per-channel reassembly state machine (no clock, no IO): in-order apply, duplicates dropped + counted, a gap synthesizes a local `gap` terminal, everything after a terminal drops, `abandon()` synthesizes the `abandoned` terminal when the consumer's timer fires. `StreamEvent::{Body, Finished, Failed}`. The synthetic finish reasons never ride the log.
- `router.rs` - `Router::{To, Broadcast, ToCapable, AllCapable}`: `To`/`Broadcast` set/clear `target_agent_id` via `apply`, the capability variants resolve against the registry via `resolve_targets(registry, now)` (one for `ToCapable` by `RoutePolicy::{Cheapest,Fastest,LeastLoaded,Sticky,Any}`, every capable agent for `AllCapable`, `LaserError::NoCapableAgent` when none). `CapabilitySelector{skill, policy}`. `InboxRoute::{Advertised, Fixed(topic)}` turns a resolved target into the topic to address: `Advertised` resolves the agent's live-presence inbox (`LaserError::NoInbox` if none, never a shared name), `Fixed` uses a caller-owned topic. A route resolves a topic within the caller's stream, the tenant boundary, cross-stream addressing is deferred federation.
- `registry.rs` - `AgentRegistry` (built by `Laser::agent_registry`): folds the card registry topic (`refresh` -> latest `RegisteredCard` per agent, `resolve(skill, now)` for the fresh capable cards) and, query-gated, the live connection presence (`refresh_presence` -> `inbox_for(agent)`). **Cached per data-stream on the connection:** `agent_registry` seeds the folded cards, quarantine set, presence, and cursor offsets from a shared `RegistryCache` (per-stream `Arc<Mutex<..>>`) and resumes the fold instead of re-reading the topic from offset 0 every call; `refresh` folds only new records, evicts cards past their `ttl_micros` (so the cached set stays bounded by live agents, not every agent ever seen), and writes the result back. Presence is cached with a ~2s ttl: `refresh_presence` reuses the last paged connection table within the ttl instead of paging every send. `Laser::advertise_presence`/`clear_presence` (query-gated) set/clear this connection's `AgentPresence` (`AGDX_SET_CLIENT_METADATA`). `Laser::client_metadata()` (query-gated) is the paginated discovery read. **Presence is principal-bound:** a folded `PresenceEntry` carries the connection's authenticated `user_id`. `inbox_for_principal(agent, user_id)` resolves the inbox only when it matches that principal (defense against a connection claiming another principal's agent id); `inbox_for(agent)` stays the claim-only resolve. **Quarantine:** `Laser::quarantine(operator, agent)` appends a `status`/`quarantine` fact to the registry topic; the fold (`apply_quarantine`) marks it and `resolve` excludes it (`is_quarantined` reports it), so a quarantined agent is never routed work. `Laser::unquarantine(operator, agent)` appends a `status`/`unquarantine` fact that lifts it, so quarantine is reversible without waiting on retention expiry. Authorized by the registry topic write ACL. **Signing (`sign` feature):** `quarantine_signed`/`unquarantine_signed` carry an ed25519 signature, and a registry whose connection enrolled a verifier (`LaserBuilder::verifier(Arc<KeyRegistry>)`) folds a quarantine/unquarantine fact only when its signature verifies; without a verifier the topic write ACL is the gate (the baseline). Identity is a claim and the broker ACL is the primary authorization, so the verifier is defense in depth over it.
- `contract.rs` - `Laser::contract(Router) -> ContractBuilder` (`from`/`payload`/`inbox_route`/`reply_on`/`expire_if_not_consumed`/`deadline`/`send`): a directed request to one agent (`to`/`to_capable`, never broadcast) bundling inbox resolution, the expiry (stamped as `deadline_micros`), and the completion deadline, returning `Contract::{Completed, Failed, NotConsumed, TimedOut}`. Sends an AGDX command (`.from(source)` required) and runs its own watch (`AgentReplyReader::poll_contract`, tail-seeded) for the `Working` ack (envelope correlation) and the terminal (AGDX response/error or a `send_agent` reply by echoed idempotency key, so the command sets `idempotency_key = correlation`). Two timers: no ack by the expiry is `NotConsumed`, no terminal by the deadline is `TimedOut`. `ContractBuilder::conversation(ConversationId)` pins the conversation the command rides: the fence gate buckets its high-water mark by conversation, so a fenced task that may be re-dispatched must use a stable conversation or each attempt lands in its own bucket (default is a fresh minted conversation per send, correct for a one-shot, non-fenced command). With a verifier enrolled, the contract reply path verifies the reply's signature before accepting a terminal, so a forged reply echoing the correlation cannot short-circuit the contract.
- `workflow.rs` - `Laser::workflow(name) -> Workflow` (`budget`/`inbox_route`/`step(label, Router, StepFn)`/`run`), with `StepHandle` (`after`/`verify_with`/`exclusive`/`compensate_with`/`on_timeout`) refining the last step. `run()` topo-sorts by `after` (cycle/unknown-dep errors), dispatches each To/ToCapable step via `Laser::contract` threading prior outputs through `StepContext`, enforces `Budget` (tokens summed from reply `usage`, wall-clock, invocations), verifies, and runs compensations in reverse on failure (saga). `StepFn`/`Verifier` are blanket-impl'd for closures. An `all_capable` step scatters a contract to every capable agent concurrently and folds the completed replies (each body on its own line) into the step output, succeeding when at least one completes (partial-panel tolerance). An `exclusive()` step claims a fenced lease on the run (`acquire_fence`, kv-gated, namespace `agdx.workflow.fence`) and stamps the lease's monotonic token on its command (`ContractBuilder::fence` -> the `agdx.fence` metadata the consumer high-water gate reads), pinned to a conversation derived from `run_id`/label so every (re)dispatch of the step shares the fence bucket (`ContractBuilder::conversation`) and the gate actually compares a stale holder's token against the new one. That gate is the local same-holder replay gate (per-conversation, per-consumer-process); the durable cross-holder at-most-once EXTERNAL effect is the handler's, by committing through `Kv::cas_fenced` carrying the token. It must be directed; without the managed KV it errors `Unsupported`. `acquire_fence` also fails closed (`Unsupported`) when the server does not advertise the fenced compare-and-swap capability (`KV_CAS_FENCED`), so the SDK refuses to claim at-most-once on a server whose fence may not be monotonic. Broadcast steps error `Unsupported` (no gather target). `StepHandle::on_timeout(OnTimeout::{Fail,Reassign})` (default `Fail`): `Reassign` (requires an exclusive step) hands a timed-out task to a fresh holder by re-acquiring the lease, which bumps the plane's fence sequence so the new token is strictly greater and the timed-out holder is gated out, bounded to `MAX_REASSIGNMENTS`. A non-exclusive step with `Reassign` is rejected at `run`. **Journal + replay:** each step's outcome is journalled to `AgentTopic::WorkflowJournal` keyed by the run's conversation, and `Workflow::run_id(id)` resumes a run by replaying the journal (re-derives recorded outputs, re-dispatches only unfinished steps); `WorkflowOutcome::run_id` is the resume handle. `AgentMessage::body()` returns the task body for both AGDX-command and `send_agent` shapes, so a step/contract worker need not know how it was reached.
- `session.rs` - `SessionPolicy::{PerCall, PerUser}`. `PerUser` -> `derive`.
- `state.rs` - `ConversationState::load(laser, conversation, topics, init, fold)`.

## Rules specific to this area

- Do not reimplement polling/commit. Wrap a handler and delegate to `IggyConsumerMessageExt::consume_messages`.
- Returning `Ok(())` from the wrapper is what lets `AutoCommitAfter` advance the offset. Skip (dedup/deadline) and dead-letter paths return `Ok(())` on purpose.
- Anything that takes a topic and needs an `Identifier` on a poll/get path uses `AgentTopic::as_identifier()`. Produce/consumer-group calls take the `&str` name (`topic_string()`).

## Review smells

- A lock held across `.await` (especially the producer map or dedup).
- `request` rescanning from offset 0 each poll instead of advancing a cursor.
- A spawned task whose error is dropped (use `AgentHandle::join`/`shutdown`).
- Dropping `AgentHandle` and expecting the agent to stop (it does not, that is by design - call `shutdown()`).
- Undecodable messages silently skipped instead of dead-lettered.
- A permanent / bad-input failure retried `max_attempts` times instead of returning `LaserError::rejected(..)` (immediate DLQ).
- Assuming a premium capability is present without checking `Laser::capabilities()`.
