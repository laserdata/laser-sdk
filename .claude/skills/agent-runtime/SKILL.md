# Agent runtime

`agent/` is the customer-facing runtime: how you send, consume reliably, route, and spawn agents. Load [laser-sdk-overview](../laser-sdk-overview/SKILL.md) first. Repo rules in [AGENTS.md](../../../AGENTS.md).

Feature `vsr` changes only the underlying Iggy client framing. Agent and AGDX publish, poll, consumer-group, replay, retry, and dead-letter paths use standard Iggy commands and keep the same API and delivery rules. Managed operations used by presence, registry enrichment, fenced workflows, and the run registry stay capability-unavailable until upstream VSR accepts AGDX's custom command codes.

## STOP and ask the user before

- Changing public signatures: `Laser::{send_agent, request, bootstrap, spawn_subconversation}`, `AgentHandler::handle` (takes `&AgentMessage` + an `&AgentCtx<'_>`), `AgentCtx`, `Agent`/`AgentHandle`, `ReliableConsumer::run`. These are the API customers code against.
- Changing the commit mode. `Serial` uses `AutoCommit::After(ConsumingEachMessage)`, and `SerialPerPartition` uses `AutoCommit::Disabled` and stores each offset explicitly only after its lane handled the message. Both keep delivery at-least-once + commit-after-handle - do not weaken that.
- Changing dedup-before-handle ordering or the DLQ behavior.

## Key files and symbols

- `laser.rs` - `Laser` facade.
  - `send_agent`: encode provenance -> headers, partition by `provenance.partition_key()`, send via a **per-topic cached producer**.
  - Producer cache: `DashMap<String, Arc<OnceCell<Arc<IggyProducer>>>>`. The shard lock is held only to fetch/insert the cell. `producer.init().await` runs inside the `OnceCell`, off the map. Never hold any map lock across the network init - that would serialize unrelated sends behind one slow topic.
  - `request`: rides the **shared reply dispatcher** (`ReplyHub`, `agent/replies.rs`), not a per-call reader. Correlation is a fresh `correlation_id` (Ulid, the `agdx.corr` header), stamped on the request and echoed back by `AgentCtx::respond` (`reply_provenance` echoes `correlation_id`, NOT the business `idempotency_key`, which stays the caller's own dedup key). `conversation_id` alone is NOT used for correlation, so a forged reply that guesses the conversation id cannot hijack the request. The hub reads each reply topic ONCE for all waiters (`subscribe(correlation)` registers before send, `ticket.wait(timeout)` completes from the dispatcher), so N concurrent requests / a fan-out's N branches share one read stream instead of N. The AGDX contract/workflow terminal-watchers keep their own tail-seeded `AgentReplyReader` (`poll_contract`/`next_agdx_match`, the ack+terminal+signature shape) for now.
  - `spawn_subconversation(&Provenance) -> Provenance`: fresh `conversation_id`, `parent_conversation_id` = parent's, `root_conversation_id` carried through.
  - `bootstrap`/`ensure_topic`/`ensure_stream`: idempotent create-if-missing. `bootstrap` also warms a producer per well-known topic concurrently.
- `consumer.rs` - the public `ReliableConsumer::run` (renamed from `AgentConsumer`, the grammar close-out: the name says what it gives you) + the private per-message `ReliableWorker` adapter (impls Iggy's `MessageConsumer`). Per message: decode (undecodable -> `dead_letter_undecodable`, `DeadLetterReason::DecodeFailed`, payload verbatim). **Target-agent filter** (skip if `provenance.target_agent_id` is set and does not match this consumer's `agent` id, defensive guard for the one-agent-one-group invariant documented on `Agent`). Dedup on `idempotency_key` via the `Deduplicator` seam, deadline check -> DLQ, then the handler with retry/backoff. The per-task fence high-water map is bounded: idle entries are swept past a ttl once it grows over a soft cap, so a long-lived consumer's map stays bounded by its active working set. A handler error that `is_retryable()` retries to `max_attempts` then DLQs. A `LaserError::Rejected` (from `LaserError::rejected(..)`) DLQs immediately, no retry. Every DLQ path publishes a typed `AgentDeadLetter` capsule (via `dead_letter`/`dead_letter_undecodable` -> `publish_dead_letter`): the `DeadLetterReason` (`DecodeFailed`/`DeadlineExceeded`/`Rejected`/`RetryExhausted`), the attempt count, the poison message's full `LogPosition` (built from `stream_id`/`topic_id` resolved once in `run`), and the original payload verbatim. The capsule message keeps the original provenance headers (deadline cleared, so a deadline-bound DLQ consumer does not re-drop it) and is marked `agdx.ct = cbor`. DLQ publication is best-effort: a failed publish is logged at `error!` with the reason and the offset still commits, so the loss is loud, not silent, and the capsule plus its publish result are handed to the optional `on_dead_letter` `DeadLetterSink` (`async_trait`, `dyn`-safe) so a lost message is an observable event, not just a log line. The handler is wrapped by an ordered `middleware: Vec<Arc<dyn AgentMiddleware>>` seam (`before_handle` once before the retry loop, a rejection dead-letters without running the handler, `after_handle` after every attempt with its result and one-based number) for auth/metrics/tracing. A deterministic wiring/startup/config failure is `LaserError::HandlerConfig` (never retryable, unlike the retryable `Handler`), so it dead-letters at once instead of burning the retry budget. `run` fires a `ready` sender once joined, and its scheduling follows `ConcurrencyPolicy` (poll interval default 10ms `POLL_INTERVAL`, overridable): **`Serial`** (default) rides `IggyConsumerMessageExt::consume_messages` with `AutoCommit::After(ConsumingEachMessage)`, one message at a time across all assigned partitions. **`SerialPerPartition { max_partitions }`** drives the `IggyConsumer` stream itself with `AutoCommit::Disabled`, routing each message to a per-partition worker lane (unbounded mpsc + spawned task, so `ReliableWorker` owns a `Laser` clone and is `Arc`-shared, `H: Send + 'static`) and committing each offset with an explicit `store_offset(offset, Some(partition))` only after its lane handled the message (matching what iggy's own auto-commit does per received partition). Lanes run concurrently across partitions, serial within one, with lane-local retry backoff, so a poison message on one partition never stalls another (a partition beyond the cap is handled inline). Shutdown is a **graceful drain**: `Serial` signals iggy's own loop to stop polling (its handler runs inside the message arm, so the in-flight message finishes) and `SerialPerPartition` drops the lane senders so each lane finishes its queue, both bounded by `shutdown_grace` (default 30s, `LaserError::Timeout` past it). `AgentHandle::abort()` stays the unconditional hard stop.
- `agent/laser.rs` - `Laser::redrive_dead_letter(&AgentDeadLetter)`: reads the original record at the capsule's `source` log position (numeric stream/topic ids), then republishes it verbatim (headers + body via `user_headers_map`), re-keyed by conversation, to its source topic. Errors (`LaserError::Invalid`) if the source stream/topic is gone or the record aged out of retention.
- `AgentMessage::from_received` uses `message.header.offset` (the message's own offset), NOT `received.current_offset` (the partition high-water, shared across a polled batch). Gotcha worth keeping: the high-water would give every message in a batch the same offset.
- **Envelope-aware consume.** `from_received` detects an AGDX message via the `agdx.av` header (`is_agdx`), decodes the `AgentEnvelope`, and exposes it on `AgentMessage.envelope: Option<AgentEnvelope>` (a decode failure dead-letters as `DecodeFailed`). The runtime `provenance` is then *synthesized from the envelope* (`provenance_from_envelope`: conversation/source/target/idempotency/deadline), so the target filter, dedup, and deadline checks read one shape for AGDX and `send_agent` messages alike. Agent ids are name strings on both sides, so `source`/`target` map straight across (a name the SDK validator rejects just drops out). Non-AGDX messages keep `Provenance::try_from` and `envelope = None`. The decoded envelope carries the `must_understand` u64 bitset: a receiver calls `envelope.unmet_requirements(understood)` (where `understood` is the OR of the `agent::features` bits it implements) and rejects/dead-letters a message with any unmet bit rather than mis-handling a feature it lacks. No feature bits are defined yet, so today every well-formed message has `must_understand == 0` and the check is a no-op awaiting the first strict-handling feature.
- `Provenance::try_from(&IggyMessage)` matches keys first and fails-not-skips: a known provenance key with a non-string value is a decode error, while foreign/unknown keys are ignored. The AGDX typed headers (`agdx.ct` u8, `agdx.av` u32, the `Uint128` routing duplicates) coexist on one record because they are foreign to the provenance dictionary, not because non-string values are skipped.
- `Deduplicator` seam (`async fn observe(key) -> bool`, `async_trait`, `dyn`-safe): default `SlidingWindow` (in-memory bounded set over the private `DedupWindow`). A durable backend drops in via the optional `deduplicator` on `Agent::builder` / `ReliableConsumer` - back it with any `StateStore` (`get`/`set`/`delete`). The managed `Kv` implements `StateStore`, so `laser.kv(ns)` is the durable drop-in (see the concierge's resolver, which uses a KV dedup). Warm dedup pre-fills whichever impl is configured. Async + dyn so a premium backend can do I/O.
- `Laser::capabilities()` -> `Capabilities` (`capabilities.rs`): the premium set the connected streaming infrastructure advertises. Defaults to `OPEN` (all false) on raw Apache Iggy. A managed runtime injects the negotiated set via `Laser::with_capabilities(..)`. Reserve premium branches behind it. Never assume premium is present. `Capabilities` groups the surfaces by what they depend on rather than a flat list: `managed` is the root switch, and `query`/`kv`/`graph`/`forks`/`a2a_gateway` nest their own sub-flags (`capabilities.kv.available`, `capabilities.kv.cas`, `capabilities.query.consistency`, ...) under it. The concierge's and governance's cloud-gated phases show the pattern (`examples::cloud_feature_ready`, gated on `capabilities.managed` or `capabilities.authz`).
- `builder.rs` - `Agent::builder` (id/listen_on/handler/optional respond_on/optional `poll_interval`/`inbox_route` defaulting to `InboxRoute::Advertised`/`capabilities`, plus the production-runtime knobs threaded through to `ReliableConsumer`: `dedup_window`, `retry(RetryPolicy)`, `verifier`, `signing_key` (the inbound signature gate and automatic pickup/terminal signer, `sign` feature), `shutdown_grace`, `concurrency(ConcurrencyPolicy)`, `middleware(Vec<Arc<dyn AgentMiddleware>>)`, `on_dead_letter(Arc<dyn DeadLetterSink>)`, `governor((Arc<dyn ActionGovernor>, GovernorMode))` re-scoping the agent's `Laser` at spawn so everything the handler publishes is governed) -> `spawn(Laser) -> AgentHandle`. The `inbox_route` sticks as the agent's routing convention and flows builder -> consumer -> ctx for the directed-send and fan-out helpers. When `capabilities` is non-empty the spawned agent self-advertises before consuming (the free `advertise` fn): publishes a non-expiring capability card to the registry (`Laser::publish_card`) and, where the server serves it, advertises its `listen_on` topic as live presence (`advertise_presence`, query-gated, best-effort except `PresenceConflict`). One connection may advertise one agent. A second identity fails rather than overwriting the first. Empty `capabilities` keeps the prior private-agent behavior. `ack_on_pickup` (off by default) makes the reliable consumer emit a `Working` task status on `respond_on` the instant an AGDX command is picked up, before the handler, so a `Laser::contract` caller can distinguish consumed from expired-unconsumed. `AgentHandle`: `ready()` waits until the agent has joined its group and is polling (await it before publishing so nothing races the join, idempotent), `shutdown()` is a **graceful drain** (finishes the in-flight message bounded by `shutdown_grace`, then returns) / `join()` surface `LaserError`, `abort()` is the immediate hard stop. **dropping it keeps the agent running** (detached), matching the old behavior - only an explicit `shutdown()` (an `Ok(())` on the channel) stops it.
- `../govern.rs` - effect-boundary policy. `QuorumGovernor` runs voters concurrently. Every mandatory voter must affirm. Mandatory errors, empty sets, invalid thresholds, duplicate names, and conflicting body replacements block. Non-mandatory errors abstain. `SwappableGovernor::swap` returns the replaced policy and `current` reads the active one.
- `../swarm.rs` - replay-safe supervisor fold. `observe` drops unattributed evidence and deduplicates by `decision_id`. Latest evidence is selected by `(at_micros, decision_id)`.
- `../crash_context.rs` - combines already-read journal, dead letter, and policy evidence. `summarize` bounds and escapes untrusted text so payload control characters cannot forge lines.
- `../intent.rs` - SDK typed records, not AGDX wire. `Intent::builder().build`, `Vote::cast`, and `decide` are fallible. Construction and replay validate voter sets, thresholds, deadlines, and body digest. Ballots are time-bounded and canonicalized, every mandatory voter must allow, and `Decision::authorizes` verifies id, digest, and policy version before an effect. Names remain claims unless signing or topic ACLs authenticate authorship.
- `ctx.rs` - `AgentCtx<'a>` handed to `handle`: `respond` (reply on `respond_on`, chained + routed to sender), `reply_on` (chained reply on an explicit topic), `send`, `request`, `respond_input` (resolve the human-in-the-loop interrupt being handled: a `response` carrying the handled command's interrupt correlation, the responder half of `Agdx::request_input`), `fan_out(selector, payload, GatherPolicy, deadline) -> Gather` (resolve every capable agent, route each branch to its own inbox via the ctx `InboxRoute`, gather replies on the orchestrator's `respond_on`. a target with no resolvable inbox is a surfaced branch failure, never a shared-topic fallback), `approval_gate(reply_topic, prompt, timeout) -> Vec<u8>` (pause the handler on a human decision: an interrupt on the human-input topic, awaiting the approver's reply, `Rejected` on rejection. A thin wrapper over `Agdx::request_input`), `spawn_subconversation`, `laser`/`message` accessors. It is how handlers act without storing `Laser`. The consumer derives `ctx.agent` from the group name and carries the builder's `respond_on` + `inbox_route`. `fan_out` and `approval_gate` are bound in Python too (`PyAgentCtx` in `foreign/python/src/agent_runtime.rs`), each constructing a transient `laser_sdk::testing::agent_ctx` scoped to the call, since PyO3 classes cannot hold the borrowing `AgentCtx<'a>` at rest.
- `agdx.rs` - the typed AGDX producer: `Laser::agdx(topic, source, conversation) -> Agdx` with the verbs `command`/`respond`/`emit`/`status`/`fail` (each returning an `AgdxSend` builder that validates the envelope at `send()`), `request_input(reply_topic, prompt, timeout)` (human-in-the-loop interrupt/resume: a prompt `command` under a fresh minted interrupt correlation, then awaits the correlated `response` on `reply_topic` and returns its body, or `LaserError::Rejected` when the reply is an `error` terminal - composes existing verbs, no new wire), and `stream(correlation, purpose) -> AgdxStream` (auto-incremented `sequence`, the purpose + abandonment bound on the opening chunk, one terminal via `finish` or `fail`. opt-in `.buffered(max_chunks, linger)` holds assembled chunk messages and appends them as ONE batch incl. the terminal, linger checked at write with no background task, sequences assigned at write so ordered reassembly on the consumer side is untouched. `Agdx::assemble` is the single envelope-to-message assembly the per-send and buffered paths share, and `topic.batching()` in `sdk/src/batching.rs` is the general-record sibling: max_records/max_bytes/linger, one partition key per handle enforced at build, inline flush on a size bound so backpressure lands on the sender, explicit `flush()`/`close()`, best-effort logged drop-flush). Every send stamps `agdx.av` (u32), `agdx.ct` (u8), the conversation as a typed `Uint128` header and the target as the agent's name string, and partitions by the conversation's canonical base32 form. `AgdxStream::write` rejects bodies above `MAX_CHUNK_BODY_BYTES` before publish and increments `sequence` only after the append succeeds. The draft chunking guidance consts (`DEFAULT_CHUNK_FLUSH_BYTES`, `DEFAULT_CHUNK_LINGER_MS`, `MAX_CHUNK_BODY_BYTES`) live here.
- `assembler.rs` - `ChunkAssembler`, the pure per-channel reassembly state machine (no clock, no IO): in-order apply, duplicates dropped + counted, a gap synthesizes a local `gap` terminal, everything after a terminal drops, `abandon()` synthesizes the `abandoned` terminal when the consumer's timer fires. `StreamEvent::{Body, Finished, Failed}`. The synthetic finish reasons never ride the log.
- `router.rs` - `Router::{To, ToPrincipal, Broadcast, ToCapable, AllCapable}`. `ToPrincipal` and `CapabilitySelector::principal(PrincipalId)` require the selected live presence to carry that server-authenticated principal, a missing or foreign claim is `RoutePrincipalMismatch` (Forbidden, non-retryable). Capability policies rank only the principal-filtered candidates. `InboxRoute::{Advertised, Fixed(topic)}` turns a resolved target into the topic to address: `Advertised` resolves live presence, `Fixed` uses a caller-owned topic. A route resolves within the caller's stream, cross-stream addressing is deferred federation.
- `registry.rs` - `AgentRegistry` (built by `Laser::agent_registry`): folds the card registry topic (`refresh` -> latest `RegisteredCard` per agent, `resolve(skill, now)` for the fresh capable cards) and, query-gated, the live connection presence (`refresh_presence` -> `inbox_for(agent)`). **Cached per data-stream on the connection:** `agent_registry` seeds the folded cards, quarantine set, presence, and cursor offsets from a shared `RegistryCache` (per-stream `Arc<Mutex<..>>`) and resumes the fold instead of re-reading the topic from offset 0 every call. `refresh` folds only new records, evicts cards past their `ttl_micros` (so the cached set stays bounded by live agents, not every agent ever seen), and writes the result back. Presence is cached with a ~2s ttl: `refresh_presence` reuses the last paged connection table within the ttl instead of paging every send. `Laser::advertise_presence`/`clear_presence` (query-gated) set/clear this connection's `AgentPresence` (`AGDX_SET_CLIENT_METADATA`). `Laser::client_metadata()` (query-gated) is the paginated discovery read. **Presence is principal-bound:** a folded `PresenceEntry` carries the connection's authenticated `user_id`. `inbox_for_principal(agent, user_id)` resolves the inbox only when it matches that principal (defense against a connection claiming another principal's agent id). `inbox_for(agent)` stays the claim-only resolve. **Quarantine:** `Laser::quarantine(operator, agent)` appends a `status`/`quarantine` fact to the registry topic. The fold (`apply_quarantine`) marks it and `resolve` excludes it (`is_quarantined` reports it), so a quarantined agent is never routed work. `Laser::unquarantine(operator, agent)` appends a `status`/`unquarantine` fact that lifts it, so quarantine is reversible without waiting on retention expiry. Authorized by the registry topic write ACL. **Signing (`sign` feature):** `quarantine_signed`/`unquarantine_signed` carry an ed25519 signature, and a registry whose connection enrolled a verifier (`LaserBuilder::verifier(Arc<KeyRegistry>)`) folds a quarantine/unquarantine fact only when its signature verifies. Without a verifier the topic write ACL is the gate (the baseline). Identity is a claim and the topic write ACL is the primary authorization, so the verifier is defense in depth over it.
- `contract.rs` - `Laser::contract(Router) -> ContractBuilder` resolves one target and watches pickup plus terminal. With a verifier enrolled, plain, invalid, and wrong-identity replies are ignored. A principal-bound route binds verification to the same authenticated principal, a claim route binds to the target's enrolled identity. Accepted replies carry `AgentMessage::verified_principal`, and scatter preserves it per branch. Python keeps `contract`/`scatter` as body-only conveniences and exposes identity through `contract_report`/`scatter_report`.
- `workflow.rs` - `Laser::workflow(name) -> Workflow` (`budget`/`inbox_route`/`step(label, Router, StepFn)`/`run`, plus `.registered()` behind the `runs` feature: gates on the `agent_workflow` capability before any publish, submits to the managed run registry converging on the run's conversation id, emits the run's lifecycle as `task` status records stamped with the pinned `run` metadata key via the shared `mark_run` helper (Responses topic, `detail` metadata on failure), reads the run row at every step boundary and routes a recorded `cancel_requested` into compensation + `LaserError::Cancelled`, and reports the terminal state), with `StepHandle` (`after`/`verify_with`/`exclusive`/`exclusive_in(namespace)`/`compensate_with`/`on_timeout`) refining the last step. `exclusive_in` pins the lease/fence sequence to a caller-chosen namespace instead of the SDK-owned default, so a handler's own `Kv::cas_fenced(key, fence_key, token)` commit (see [kv](../kv/SKILL.md)) can share and validate the same fence sequence as the workflow's reassignment gate. `run()` topo-sorts by `after` (cycle/unknown-dep errors), dispatches each To/ToCapable step via `Laser::contract` threading prior outputs through `StepContext`, enforces `Budget` (tokens summed from reply `usage`, wall-clock, invocations), verifies, and runs compensations in reverse on failure (saga). `StepFn`/`Verifier` are blanket-impl'd for closures. An `all_capable` step scatters a contract to every capable agent concurrently and folds the completed replies (each body on its own line) into the step output, succeeding when at least one completes (partial-panel tolerance). An `exclusive()` step claims a fenced lease on the run (`acquire_fence`, kv-gated, namespace `agdx.workflow.fence`) and stamps the lease's monotonic token on its command (`ContractBuilder::fence` -> the `agdx.fence` metadata the consumer high-water gate reads), pinned to a conversation derived from `run_id`/label so every (re)dispatch of the step shares the fence bucket (`ContractBuilder::conversation`) and the gate actually compares a stale holder's token against the new one. That gate is the local same-holder replay gate (per-conversation, per-consumer-process). The durable cross-holder at-most-once EXTERNAL effect is the handler's, by committing through `Kv::cas_fenced` carrying the token. It must be directed. Without the managed KV it errors `Unsupported`. `acquire_fence` also fails closed (`Unsupported`) when the server does not advertise the fenced compare-and-swap capability (`KV_CAS_FENCED`), so the SDK refuses to claim at-most-once on a server whose fence may not be monotonic. Broadcast steps error `Unsupported` (no gather target). `StepHandle::on_timeout(OnTimeout::{Fail,Reassign})` (default `Fail`): `Reassign` (requires an exclusive step) hands a timed-out task to a fresh holder by re-acquiring the lease, which bumps the plane's fence sequence so the new token is strictly greater and the timed-out holder is gated out, bounded to `MAX_REASSIGNMENTS`. A non-exclusive step with `Reassign` is rejected at `run`. **Journal + replay:** each step's outcome is journalled to `AgentTopic::WorkflowJournal` keyed by the run's conversation, and `Workflow::run_id(id)` resumes a run by replaying the journal (re-derives recorded outputs, re-dispatches only unfinished steps). `WorkflowOutcome::run_id` is the resume handle. `AgentMessage::body()` returns the task body for both AGDX-command and `send_agent` shapes, so a step/contract worker need not know how it was reached. The wire `RunBudget` is distinct from this coordination `Budget`: `Runs::submit_budgeted` carries a multi-dimensional per-run `RunBudget` (events/model_calls/tool_calls/patches/depth/wall_clock/cost), accumulated in the plane run fold, failing the run on a crossed cap, surfaced as `LaserError::BudgetExceeded`.
- `session.rs` - `SessionPolicy::{PerCall, PerUser}`. `PerUser` -> `derive`.
- `../testing.rs` (`laser_sdk::testing`, `agent` feature) - handler unit-test seam: `agent_message(payload, provenance)` builds a plain `AgentMessage`, `agent_ctx(&laser, &message, agent, respond_on, inbox_route)` builds a borrowed `AgentCtx` over caller-owned locals (no live consumer). A handler that only reads its message needs no server, and one that calls ctx IO helpers points `laser` at a test server. Bound in Python as module-level `agent_message`/`agent_ctx` functions (`foreign/python/src/agent_runtime.rs`), returning an owned `AgentMessage`/`AgentCtx` since Python has no borrow to preserve.
- `state.rs` - `ConversationState::load(laser, conversation, topics, bound, init, fold)` under an explicit `ReplayBound`, plus `load_with(store, ..)` seeding from a `SnapshotStore`. See [context-and-memory](../context-and-memory/SKILL.md).

## Rules specific to this area

- The `Serial` path wraps a handler and delegates to `IggyConsumerMessageExt::consume_messages` (do not reinvent it). The `SerialPerPartition` path deliberately drives the `IggyConsumer` stream itself with `AutoCommit::Disabled` and explicit per-partition `store_offset` - the ONE place polling/commit is hand-rolled, because iggy's single-loop `consume_messages` cannot process partitions concurrently. Keep commit-after-handle in both.
- Returning `Ok(())` from the wrapper is what lets `AutoCommitAfter` advance the offset. Skip (dedup/deadline) and dead-letter paths return `Ok(())` on purpose.
- Anything that takes a topic and needs an `Identifier` on a poll/get path uses `AgentTopic::as_identifier()`. Produce/consumer-group calls take the `&str` name (`topic_string()`).

## Review smells

- A lock held across `.await` (especially the producer map or dedup).
- `request` rescanning from offset 0 each poll instead of advancing a cursor.
- A spawned task whose error is dropped (use `AgentHandle::join`/`shutdown`).
- Dropping `AgentHandle` and expecting the agent to stop (it does not, that is by design - call `shutdown()`).
- Undecodable messages silently skipped instead of dead-lettered.
- A permanent / bad-input failure retried `max_attempts` times instead of returning `LaserError::rejected(..)` (immediate DLQ).
- Assuming a premium capability is present without checking `Laser::capabilities()`.
