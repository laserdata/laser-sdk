# Context and memory

These modules reconstruct state from the durable log: assemble an ordered window of a conversation, fold it into state, or recall scoped memory items. Load [laser-sdk-overview](../laser-sdk-overview/SKILL.md) first. Repo rules in [AGENTS.md](../../../AGENTS.md).

Feature `vsr` changes only the underlying Iggy client framing. Context assembly, cursors, conversation-state replay, topic snapshots, and folded `LogMemory` use standard append and poll commands and keep the same behavior. KV-backed recall, query, graph, and other managed memory composition remains unavailable until upstream VSR accepts LaserData's custom command codes. Callers must keep the existing capability and `Unsupported` handling.

## STOP and ask the user before

- Changing the cross-topic ordering key in `ContextAssembler::assemble` (`(timestamp, topic_index, partition, offset)`) - it defines what an LLM is fed.
- Changing the `MemoryLogEntry` framing (the typed `Item`/`Forget`/`Feedback` records `LogMemory` writes to the audit topic) - it is the wire format of a memory record on the log.
- Changing how `recall` scopes results (agent filter, multi-tenancy is at the Iggy stream boundary, not a memory scope field).

## Key files and symbols

- `poll.rs` - `drain_partition(client, stream, topic, consumer, partition, from_offset, batch) -> PartitionBatch { messages, next_offset }`. The single shared batched-drain used by both context and the reply scanner. Reuse it, do not hand-roll another poll loop.
- `context.rs` - `ContextAssembler` (bon builder: `conversation_id`, `across_subconversations`, `topics`, `policy`). `assemble` reads each topic's partitions via `drain_partition`, filters by `matches` (conversation, or root/parent when `across_subconversations`), orders by Iggy timestamp with a deterministic tie-break, then applies the `ContextPolicy`. Policies: `LastN`, `RoleFilter`. Read paths use `AgentTopic::as_identifier()`. NB: `assemble` is **stateless** - it uses a throwaway consumer and reads each partition from offset 0 on every call (`drain_partition(.., from_offset: 0, ..)`), so cost scales with topic size. That is a choice, not a limit: a reader that committed a consumer offset and folded incrementally reads only new messages - which is exactly what `LogMemory` (below) and the public `Cursor` (`cursor.rs`) now do. Keep this in mind before claiming any log-backed recall is inherently O(history).
- `memory.rs` - `Memory` trait (the four verbs `remember`/`recall`/`improve`/`forget`) with one durable model (`LogMemory`, the default) plus the in-process `VectorMemory` for similarity (semantic vs recency, so they are not Liskov-substitutable). Shared vocabulary lives here too (not the wire crate): `MemoryKind` (incl. `Procedure`, with `.class()` -> `MemoryClass::{Episodic,Semantic,Procedural}`, the field's taxonomy), `Lifetime`, `RecallStrategy` (incl. `Keyword` lexical signal and `Hybrid` multi-signal fusion), `Feedback`, `MemoryScope` (identity layers `user`/`agent`/`conversation`(session)/`app` over the physical `stream`), and `MemoryId` (a ULID by default, or content-addressed via `MemoryId::content`, which routes through the wire crate's one canonical `laser_wire::hashing::content_id` so the dedup id matches cross-SDK). Three seams sit alongside `Embedder`: `Reranker` (+ `RerankedMemory<M,R>` wrapper, the optional rerank second stage of recall), `Summarizer` (session bodies in, summary body out, the consolidation summarize pass, the SDK ships no model client) and `Consolidator` (+ `ConsolidationReport`, the async "memify"/sleep-time pass). Type-erased shared forms (`SharedEmbedder`/`SharedReranker`/`SharedConsolidator`, plus `MaybeEmbedder` erroring with the registration hint only when a strategy needs an embedding) let `MemoryHandle` own the seams without generics. `RecallSignal` + `MemoryItem.signals` carry per-item attribution (strategy, rank, pre-fusion score) and `fuse_reciprocal_rank` is the pure RRF fusion Hybrid rides. There is **no** memory wire band: memory is a client composition over publish + query + graph (see [wire-contract](../wire-contract/SKILL.md) and the memory chapter of `docs/agdx.md`).
  - `LogMemory` (log-backed on `AgentTopic::Audit`): `remember` writes a typed `MemoryLogEntry::Item { id, kind, body }`, `improve` writes `MemoryLogEntry::Feedback { target, weight }`, `forget` writes `MemoryLogEntry::Forget { target }`. The entry is named-field CBOR (`laser_wire::framing`), not a magic-prefixed payload. **KV-backed recall (the default)**: `recall`/`fetch_named` read the managed key-value read view the deployment materializes from the memory topic, so a large topic never folds to answer one recall. `recall_kv` scans the namespace narrowed to the conversation (the conversation lens), decodes each `KvEntry.scope` into a `MemoryItem` (`MemoryKind::from_word`, agent and conversation parsed into the typed wrappers at the SDK edge), filters by agent, and takes the most-recent by ULID id. `update_named` still folds its read for read-your-writes on a merge. Without the `kv` feature (or against raw Iggy with no read view) the default recall returns `Unsupported`. **Folding is the explicit opt-in**, never the default: `RecallBuilder::folded()`, `MemoryHandle::recall_folded`/`fetch_folded`, `LogMemory::recall_folded`/`fetch_named_folded` hold a folded `Projection` (items + tombstoned ids + accumulated feedback) + a per-partition cursor behind a `tokio::sync::Mutex`, `catch_up` drains only new audit messages via `poll::drain_partition` (the same primitive the public `Cursor` rides), for a small serverless deployment. The fold is a pure, unit-tested fn. Isolation is at the Iggy stream level.
  - `VectorMemory<E: Embedder>`: semantic, in-memory. `remember` embeds the payload, `recall` fuses the active signals per `query.strategy` (semantic cosine + lexical `keyword_score` + accumulated `feedback`, `Keyword` skips the embed call, `Hybrid` fuses all), filters by the `MemoryScope` identity layers (conversation/agent/user/app), and falls back to recency when no signal is active. `Laser::memory_with(.., Vector)` and Python `laser.vector_memory(..)` retain that Laser's governor: remember/improve/forget decide before local mutation, item decisions see the raw proposed body, and content-addressed ids from `.dedup()` are preserved. Direct `VectorMemory::new` remains standalone, while `VectorMemory::governed` is the explicit low-level form. The `Embedder` trait is the model seam (impl lives in app code, like `LlmClient`). Wrap with `RerankedMemory` for a second-stage `Reranker`. A durable vector store backed by an external relational store is a future drop-in. Embed the query *before* taking the items lock (no model call under the lock).
  - `MemoryHandle` (the front door): one model. `Laser::memory(ns)` writes every remember/forget/feedback to a memory topic (the source of truth) and recall reads it back, a deployment materializes the topic into the versioned key-value read view. `Laser::memory_on_topic(name)` and the `Laser::memory_topic(name).stream(s).partitions(n).ttl(d)/.no_expiry().build()` builder configure the topic: stream, name, partition count (each scope keyed to one partition), and message-expiry (default `DEFAULT_MEMORY_TOPIC_TTL` = 30 days on the stream, separate from the materialized read view's own retention). `MemoryBackend` is `{Auto, Log, Vector}`: `memory_with(ns, MemoryBackend::Vector).embedder(..)` builds the in-process similarity index (for tests and offline recall), `.reranker(..)` wraps recall. Fluent verbs: `remember(payload).scope(c).kind(..).dedup().send()`, `recall(c).semantic(..)/.keyword(..)/.hybrid(..)/.strategy(..).limit(n).fetch()`, the one-call `context(conversation, token_budget)` altitude, and `consolidate(scope, max_items)`.
  - **Consolidation**: `DefaultConsolidator<M, S>::new(memory, max_items)` implements the `Consolidator` contract with the prune pass (keep the newest `max_items`, oldest-first by the ULID `MemoryId`), and `.with_summarizer(..)` adds the summarize pass (Message-kind items fold through the `Summarizer` seam into one durable summary. `.prune_summarized()` forgets the folded turns). `Agent::builder().consolidate_every(d) + .consolidator(SharedConsolidator::new(..))` runs it off the handler loop, aborted with the agent. The fact-invalidation half (same subject + predicate, different object -> `GraphHandle::unlink` closes the superseded edge bitemporally + an `improve` link) layers next, contradiction beyond same-subject-same-predicate stays the implementor's policy.
  - **Recall budgeting**: `MemoryQuery` carries an optional `token_budget` alongside `limit`, and the free `to_context_block(items, token_budget)` renders a recall result into one prompt block in rank order, filling to the budget and appending a `[..., N more omitted ...]` marker on overflow. Tokens are estimated at ~4 bytes each (`estimate_tokens`), advisory and tokenizer-free (a safe lower bound, no model dep). The first item always renders, so a non-empty result never renders empty.
  - `GraphHandle` (feature `graph`, `sdk/src/graph.rs`, `Laser::graph(name)`): the managed knowledge-graph surface, the durable relationship layer agentic memory composes over. `upsert(nodes, edges)` writes (idempotent on content-addressed ids, `GraphEdge::relate(..).valid(from,to)` stamps a bitemporal valid-time window), `neighbors(node, dir, edge_type, depth)` is the one-hop read, and the fluent `start_ids`/`start_match`/`start_nearest` + `out`/`incoming`/`both` + `return_edges`/`return_paths` + `.as_of(micros)` builder runs a multi-hop `fetch` traversal. The managed engine serves all returns (nodes/edges/paths/triplets) and starts (ids/match/nearest), `as_of` follows only edges valid at that instant. Nodes and edges are minted in the wire crate (`GraphNode::entity` / `GraphEdge::relate`, ids via `NodeId::content` / `EdgeId::content` over `content_id`, so the same entity converges cross-SDK). A node and an edge each carry an optional `SourceRef` (`GraphEdge::with_source`), the origin record an extraction came from (first-writer on the node, last-writer on the edge), excluded from the content-addressed id. The `SourceRef::Message` form carries an optional `conversation`, which is what the read-side conversation lens matches, and `.conversation(id)` on the handle (or the builder) narrows a traversal - neighbors or a multi-hop fetch - to the elements one conversation asserted. Graph projections register through `Laser::projections().register_graph(..)` / `drop_graph(..)`. All gated on the `graph` capability: against raw Apache Iggy every call returns `Unsupported`. The ops are also capability-gated by the fork-native authz layer like `kv`, an ungranted call refusing `GraphError::Unauthorized`.
- **Named-item altitude**: `MemoryHandle::{set, fetch, update, remove}(key, ..)` is point state under a caller `&str` key, and rides the memory topic like every other write - never a direct KV write. `set` publishes a `laser_wire::memory::MemoryRecord::Item { id: "<namespace>/<key>" }` (byte-identical to the recall records, so the plane's fold materializes it too), `remove` a `Forget`, `update` folds the current value + RFC-7386 merge-patches it + republishes (`merge_named`). `fetch` folds the topic (`Projection.named`, keyed by the prefixed id) so a fresh handle sees prior writes, a named id is not a ULID, so it never decodes as a recall item. The `LogMemory` handle carries the namespace prefix, the vector handle refuses typed (`named_memory`). Reused across `MemoryHandle::{set,..}` and the `LogMemory::{set_named,update_named,fetch_named,forget_named}` the python bindings call. Mirrored on python `Memory.set/fetch/update/remove` (durable handle only, UnsupportedError on the vector handle).
- `agent/state.rs` - `ConversationState::load(laser, conversation, topics, bound, init, fold)` replays the topics under an explicit `ReplayBound` (`FromOffsets(map)` incremental, `Last(n)`, or `Full` written out, never a silent default: the bounded-reads law) and folds from `init`. `ConversationState::load_with(laser, store, ..)` seeds from a `SnapshotStore` (`sdk/src/snapshot.rs`: `latest`/`save`, `KvSnapshotStore` managed point-state, `TopicSnapshotStore` log-native with a backward tail scan) and replays only from `as_of + 1` per partition (`resume_offsets`). The snapshot's `state` bytes decode as JSON into the fold's `S`. `ContextAssembler` gained `.from_offsets(map)` (per-partition start offsets) to carry the resume. `ContextScope::state(topics, bound, init, fold)` / `state_with(store, ..)` are the accessor spellings. `Laser::context(conversation)` is the conversation accessor (`append`/`fetch`/`block`/`state`), and `ContextScope::memory(namespace)` / `memory_with(ns, backend)` return a `ScopedMemory` that bakes the conversation into `recall`/`remember`/`block(budget)`/`consolidate(max)` so a session's messages and its memory share one scope, durable memory stays cross-conversation via `Laser::memory`, and `ContextScope::graph(name)` reaches the knowledge graph from the scope as a convenience but returns it unnarrowed (a dependency/knowledge graph is shared across conversations, so the scope never auto-filters it - narrow explicitly with `graph(name).conversation(id)` when you want the slice one conversation asserted). Python mirrors it: `laser.context(conv)` -> `ContextScope.append/fetch/block/memory(mem)` and `ScopedMemory.recall/remember`.
- `cursor.rs` - `Topic::replay()` -> `Cursor`: resumable, offset-addressable read over a topic. `poll` drains every partition from where it stopped (only new messages), `offsets()` exposes the per-partition cursor and `from_offsets(..)` resumes it. Offsets are caller-owned: checkpoint them into any `StateStore` to survive a restart. Yields `Message` (raw payload + headers, no `Provenance` coupling - works on any topic). This is the open primitive the `Agent` runtime (consumer groups) sits above. Reuses `poll::drain_partition`.
- `state_store.rs` - `StateStore` trait (`get`/`set`/`delete`), the one point-store seam for dedup persistence, conversation/cursor checkpoints, or arbitrary agent state. `get`/`set`/`delete` is shared with `Kv`, which **implements `StateStore`** (so the managed KV store is the durable drop-in). Self-contained defaults: `InMemoryStore` (in-mem) and `FileStore` (a local directory). `FileStore` hex-encodes keys into file names (no path traversal) and writes through a unique `<file>.<ulid>.tmp` staging file + `rename` so a crash mid-write does not leave a zero-length file and concurrent writers do not race on the same staging path.

## Rules specific to this area

- Ordering across topics is best-effort: deterministic but not strictly chronological for same-microsecond messages on different topics (Apache Iggy has independent offset spaces). State this. Do not pretend it is total order.
- `recall` honors the scope it was stored under. If you add a scope dimension, stamp it in `remember`'s provenance and filter it in `recall` - do not leave a scope field unused.
- `LogMemory.recall` returns the last `query.limit` items for the conversation. It folds the audit log **incrementally** (a maintained `Projection` + per-partition cursor, draining only new messages), not from offset 0 each call. Preserve those results-equal-to-a-full-fold semantics. The incrementality is an optimization, not a behavior change. `ContextAssembler` (one-shot folds) stays stateless by design.

## Review smells

- A second copy of the partition-drain loop instead of `poll::drain_partition`.
- A `MemoryScope` field written on `remember` but ignored by `recall` (or vice versa) - the original scope/query mismatch bug.
- Ordering code that assumes cross-topic chronological order.
