# Query layer

A queryable read-model over the log, reachable on the one Iggy connection. The client builds a structured `Query`, calls `laser.query(q)`, and gets rows back. LaserData Cloud materializes the projection and serves the query. Load [laser-sdk-overview](../laser-sdk-overview/SKILL.md) first. Repo rules in [AGENTS.md](../../../AGENTS.md).

## The boundary that matters (do not blur it)

- **The SDK ships the client only** (`sdk/src/query/`, feature `query`): the DSL, wire envelope, and `Laser::query`. Projection, binding, and schema-registry control live in `laser_sdk::projections` (feature `projections`). Streaming records, content types, codecs, and publish builders belong to `laser_sdk::stream`. An app importing the `laser-sdk` crate **never holds a backend or a query engine.**
- **The engine, projector, and every backend are managed-side** (LaserData Cloud), not in this repo. Do not add a backend, a query engine, or a `QueryBackend` trait to `sdk/src/`. That was an explicit correction.

**Generic core (done):** `Laser` lives in `src/laser.rs` (core, gated `feature="streaming"`). Public core methods: `new`, `with_capabilities`, `with_ops_stream`, `with_control_topic`, `with_dlq_topic`, `with_changes_topic`, `client`, `stream`, `ops_stream`, `control_topic`, `dlq_topic`, `changes_topic`, `capabilities`. `ensure_topic` / `ensure_topic_on(stream, ..)`, `send_with_headers` / `send_with_headers_on(stream, ..)`, and `send_batch` / `send_batch_on(stream, ..)` are crate-internal (`pub(crate)`) plumbing the public `Topic`/`publish` surface builds on, not a caller-facing API. **Query path:** `execute_query` gates on `capabilities.query.available` (probed once at `build` via `AGDX_HELLO_CODE`, the AGDX spec, folded from `capabilities.managed`, the root managed switch), encodes the `QueryEnvelope`, sends `AGDX_QUERY_CODE` via the binary transport's `send_raw_with_response` (off the log, no reply topic, no correlation poll), and decodes the returned bytes as `QueryReply`. There is no topic request/reply query path and never will be. Query is LaserData Cloud only, and against raw Apache Iggy it returns `LaserError::Unsupported`. Producers are cached per `(stream, topic)`. The ops stream defaults to `OPS_STREAM_DEFAULT` (`_agdx`) and, like the three ops-topic names (`control.commands`, `dlq`, `changes`), is overridable via the matching `LaserBuilder`/`Laser` method, but today no managed backend reads a configured name back (LaserData Cloud's plane imports the same four `laser_wire::topics` constants directly). The override is only live in `sdk/tests/integration/harness.rs`, which sets a per-test `ops_stream` for isolation on a shared raw Apache Iggy container, no plane involved. `Record` has no `Provenance` field. See `the AGDX spec` for the full wire contract. **`query` feature has no `agent` dependency**: `--no-default-features --features streaming` builds the streaming substrate alone, and `--features query` adds the managed query client without the agent runtime. The agentic layer (`Provenance`, `AgentTopic`, `Agent`, router/session/memory, `send_agent`, `bootstrap`) lives in `agent/` under the `agent` feature.

**Wire format is CBOR** (RFC 8949, ciborium). `QueryRequest`/`QueryReply` go on the wire as CBOR named maps, roughly 30-50% smaller than JSON, with native bytes and floats (the byte payload rides as a CBOR byte string, embeddings as native `f32`, not decimal strings). The user payload format is the caller's `ContentType` choice. The SDK envelope is always CBOR.

## STOP and ask the user before

- Putting a `QueryBackend`, any backend impl, or a query executor into `sdk/src/`. The SDK is transport. That was an explicit correction.
- Re-introducing a topic request/reply path for query in any form. It is permanently removed. Query is the off-log `AGDX_QUERY` managed command only.
- Changing the wire envelope (`QueryEnvelope`/`QueryReply`/`ControlEnvelope`), the ops stream / topics (`OPS_STREAM` = `_agdx`, `CONTROL_TOPIC` = `control.commands`, `DLQ_TOPIC`), `WINDOW_START` (`window_start`), or the `QUERY_OP_VERSION` / `CONTROL_OP_VERSION` versions, all on-the-wire, defined once in laser-wire and consumed by LaserData Cloud as the same crate. Aggregate results are keyed per `AggCall.alias` (the fluent helpers default the alias to the function name), not a single `agg_value` column. See `the AGDX spec`.
- Adding payload-field (Tier-2) indexing to the SDK. v1 indexes `agdx.idx.*` headers only. The payload stays opaque.

## What ships (client transport)

`sdk/src/query/` re-exports the materialized-view wire types from laser-wire and gates the managed runtime behind feature `query`. Projection/binding/schema control is owned by `laser_sdk::projections` (feature `projections`), streaming publish and record lowering by the stream layer. The query wire shapes live in `wire/src/{query,browse,control}.rs`, change them there (see [wire-contract](../wire-contract/SKILL.md)):

- **DSL** (serde plus strum, snake_case wire names, no magic strings): `Query` (`index`, `by_key`, `message_type`, `time_range`, `filter`, `vector`, `order`, `limit`, `offset`, `aggregate`, `having`, `distinct`, `select`, `fork`, `raw_sql`, `consistency`). `filter`/`having` are `Option<Filter>` where `Filter` is a recursive tree `All(Vec<Filter>)`/`Any(Vec<Filter>)`/`Not(Box<Filter>)`/`Pred(Predicate)` (externally tagged: `{"all":[{"pred":{...}}]}`), NOT a flat AND list. `Predicate`, `CmpOp`, `Sort`, `Dir`, `VectorQuery`, `Aggregate { group_by, funcs: Vec<AggCall>, window }`, `AggCall { func, field, arg, alias }`, `AggFunc` (`count`/`count_distinct`/`sum`/`avg`/`min`/`max`/`percentile`/`stddev`), `Window`, `RawSql { sql, params }`, `Select`, `Value`, `Row`, `QueryResult`. `Value` is `serde(untagged)`. Time is `u64` epoch micros (`IggyTimestamp` is not serde-able). The IR is backend-neutral: LaserData Cloud compiles it per backend and gates each construct by `BackendCapabilities`, so an unsupported construct returns `QueryError::Unsupported`, never a wrong answer.
- **Result plus pagination**: `QueryResult { rows, page: Page }`. `Row { headers: BTreeMap<String,String>, metadata: BTreeMap<String,String>, partition, offset, stream: Option<u32>, topic: Option<u32>, payload: Option<Vec<u8>>, score: Option<f32> }`. Indexed scalars ride `headers`, ride-along publisher headers (`agdx.ct`, `agdx.ref`, `agdx.sid` if set, anything custom via `.header(k,v)`) ride `metadata`, the body rides `payload` and is returned only when both `Select::payload` is set AND the publisher inlined it. `stream`/`topic` are the numeric source ids (not names, so the pointer stays compact and rename-proof), stamped per row so a reader can jump a query row back to its origin log record via `partition`/`offset`. `Page { offset, limit, total, has_more }` plus `total_pages()`. `limit`/`top_k` above `MAX_PAGE_SIZE` (1000) is **rejected** with `QueryError::TooLarge` (not silently clamped). The reply is also bounded by `MAX_QUERY_REPLY_BYTES` (64 MiB) as it builds, so a wide `with_payload` page over budget gets the same error. Page explicitly with `limit`/`offset`, walk under `.max_rows(n).rows()`, or opt into full materialization with `.fetch_all()`.
- **Consistency**: `Query.consistency: Consistency = Eventual (default) | ReadYourWrites | Strong`, absent on the wire for `Eventual`. Builder: `.read_your_writes()` / `.consistency(level)`. `ReadYourWrites` waits for the projector to reach the source log head and returns `QueryError::Stale { what, applied, required }` (`LaserError::is_stale()`) if it can't catch up in time, never a downgraded stale read. Backend-gated by the served consistency ladder (advertised on the `AGDX_HELLO` reply's `OpVersions.features` bits `READ_YOUR_WRITES` / `STRONG_CONSISTENCY`, folded into the single `capabilities.query.consistency: Consistency` field, since `Strong` subsumes `ReadYourWrites` subsumes `Eventual`). `execute_query` pre-gates locally via `Capabilities::serves_consistency(level)` (`level <= capabilities.query.consistency`): a non-Eventual level the deployment does not advertise returns `Unsupported` before the round-trip. This is mandatory, not optional, because `consistency` is an additive field an unaware backend silently ignores (it would serve an eventual read and look successful), so the SDK must refuse rather than downgrade. `Eventual` is always served, so the default path is unaffected.
- **Unified result code**: every surface error (`QueryError`/`KvError`/`ForkError`) projects onto `laser_wire::result::ResultCode` (`Ok`/`Unsupported`/`NotFound`/`InvalidArgument`/`TooLarge`/`Conflict`/`Stale`/`VersionSkew`/`Unauthorized`/`Backend`/`Unrecognized(u16)`) via `From`, and `LaserError::code()` surfaces it. `ResultCode::code()` is the pinned numeric, `http_status()` the one HTTP mapping every surface shares. Generic clients dispatch on the code, and specialists still read the typed variant.
- **Query text search**: `.text(query)` / `.text_in(field, query)` ride the additive `Query.text` IR field (relevance in `Row.score` like vector distance, text capped at `MAX_TEXT_QUERY_BYTES`). Gated on `capabilities.query.keyword` (folded from the `KEYWORD_SEARCH` feature bit) and refused locally before sending when unadvertised, since an unaware backend would silently drop the field and answer wider than asked.
- **Typed topics** (`sdk/src/typed.rs`): `laser.topic(name).json::<T>()` / `.cbor::<T>()` (serde forms, no registry) and `.schema::<T>(id).await` (resolves + compiles the registered writer schema ONCE per handle, `schema-codecs` feature). `publish(&body)` encodes, validates (schema form), and opens the ordinary `PublishRequest` with `agdx.ct` (+ `agdx.sid`) stamped, so builder verbs chain. `records(reader_name)` is the Cursor-backed typed reader (`from_offsets`/`offsets` resume, `batch` cap, `poll()` the one-poll altitude): a record that does not decode yields `TypedDecodeError { position, source }` with the position from the record's own header and the reader moves past it, `From<TypedDecodeError> for LaserError` folds into the codec family for `?`. The reliable path composes (decode in the handler, DLQ unchanged). Protobuf schema publish is refused `Invalid` (encode with prost + `.raw_bytes`), decode works. No wire change, convention over the pinned headers.
- **Change feed** (`sdk/src/watch.rs`): `laser.watch().index(name).records()?` opens a `WatchReader`, the plain `Cursor` over the `changes` topic on the ops stream. The plane publishes one `ChangeRecord { index, partition_id, from_offset, to_offset, rows }` per committed projector batch for a binding built with `.notify()`. The record is a wakeup, not truth: read the rows via `query`, resume via `offsets()`/`from_offsets()`. Gated on `capabilities.watch` (the `WATCH` feature bit), typed refusal when unadvertised. Undecodable records on the topic are skipped, index filter is client-side.
- **Produce, fluent**: `laser.topic(name).publish()` (or `laser.stream(name).topic(name).publish()` cross-stream) returns `PublishRequest<'_>`. Canonical usage is dumb-producer: `laser.topic(name).publish().json(&body)?.send().await?`. The `Projection` registered on the projector owns the schema (fields, JSON-pointer extraction, inline-payload policy, vector field), so producers do NOT stamp `.index(k, v)` per record in the canonical path. The header escape hatch remains for opaque-binary payloads the projector cannot decode: `.index(k, v)` (accumulating, capped at `MAX_INDEX_ENTRIES_PER_RECORD` = 32, key non-empty and not starting with the reserved `idx.` prefix), `.header(k, v)` (non-indexed metadata, reserved keys rejected), `.inline_payload()` (per-record override, the projection default normally suffices), `.json(&T)` / `.msgpack(&T)` / `.encode_with::<C, _>(&T)` / `.raw_bytes(bytes, ContentType::*)` / `.payload(payload)`, `.content_type` / `.projection_ref("order.v1")` / `.schema_id(u32)` (the registered writer schema for a schema-first codec, NOT routing), `.send().await`. Lowers to `agdx.ct`/`agdx.ref`/`agdx.sid`/`agdx.inline`/`agdx.idx.*`/user metadata headers. The raw-bytes-with-headers path is `topic.send(payload, headers, key)`.
- **Batch produce, fluent**: `laser.topic(name).publish_batch()` returns `BatchPublishRequest<'_>`: same batch-wide directives plus `.add_json(&T)` / `.add_msgpack(&T)` / `.add_payload(payload)` / `.add_record(payload, Record)`, plus per-record-projection variants `.add_json_with_projection(ref, &T)` / `.add_msgpack_with_projection` / `.add_payload_with_projection`. `extend_json(iter)` / `extend_msgpack(iter)` fold an iterator. `.send().await` flushes one Iggy `send_messages` call.
- **Query, fluent**: `laser.query(index)` returns `QueryRequest<'_>`: `where_eq` / `filter_eq` / `filter_ne` / `filter_gt`/`gte`/`lt`/`lte` / `filter_in` / `filter_contains` / `filter_prefix` / `filter(Filter)` (compose `Any`/`Not` trees, AND-folds into the query filter) / `conversation(id)` (the conversation lens: sugar for a `conversation_id` predicate over the field the deployment auto-projects on every row from `gen_ai.conversation.id`, no new IR field) / `message_type` / `time_range` / `order_asc`/`desc` / `limit` / `offset` / `with_payload` / `select_fields` / `distinct` / `count`/`count_distinct`/`sum`/`avg`/`min`/`max`/`stddev`/`percentile`/`agg_as` / `group_by` / `window` / `having` / `raw_sql`/`raw_sql_with` / `nearest`/`nearest_in`. Finishers: `.fetch().await -> QueryResult` (paged plus metadata), `.fetch_typed::<T>().await -> Vec<T>` (JSON serde decode), `.fetch_one::<T>().await -> Option<T>`. Codec-generic variants `.fetch_typed_with::<C, T>()` / `.fetch_one_with::<C, T>()` decode with any `Decoder`. Auto-paginating is explicit: `.max_rows(n).rows() -> QueryRows<'_>` and `.max_rows(n).rows_typed::<T>() -> TypedQueryRows<'_, T>` yield one bounded row at a time. Calling either without the ceiling is refused with `Invalid`. `.fetch_all()` / `.fetch_all_typed::<T>()` are the deliberate unbounded materialization opt-in. `Value` has `From` impls so values pass as bare literals. Drop to `laser.execute_query(Query)` for power users.
- **`Laser` itself** is `Clone` via internal `Arc<LaserInner>`. Build with `Laser::connect(connection_string)` (or `connect_with_stream(connection_string, stream)` to bind a default stream) or `Laser::builder()` (BYO client, custom credentials/capabilities/topic names). `Laser::from_client(client)` is the sync BYO entry. No `Arc<Laser>` in user code.
- **Envelope**: `QueryEnvelope{v, query}` / `QueryReply::{Ok(QueryResult)|Err(QueryError)}`, `QUERY_OP_VERSION`. `QueryError` (`Unsupported`/`IndexNotFound`/`ForkNotFound`/`Backend`/`TooLarge`/`Version`/`Stale`/`Unauthorized`) maps to `LaserError` via `From`. The plane's per-source DSL capability check refuses an ungranted read with `QueryError::Unauthorized` (`ResultCode::Unauthorized`). `TooLarge { what, size, cap }` is raised for an over-cap `limit`/`top_k` (also caught SDK-side before the round trip) or an over-budget reply, and on the HTTP edge it maps to `413`.
- **Control client**: `laser.projections().register(Projection)` / `laser.bindings().apply(ProjectionBinding)` / `laser.projections().drop(id)` / `laser.bindings().remove(SourceSelector, Option<projection_ref>)` publish a `ControlEnvelope{v, timestamp_micros, command}` (`ControlCommand::{RegisterProjection, DropProjection, ApplyBinding, RemoveBinding}`) to the control topic on the ops stream (`CONTROL_TOPIC`, `CONTROL_OP_VERSION`), co-located on one partition so register lands before binding. This is the pinned managed control-topic wire contract: it is how a client drives a real LaserData Cloud (register before publishing and querying).
- **Registry browse** (read-only): `laser.projections().get(id)` -> `Option<ProjectionInfo>` and `laser.projections().list()` -> a `ProjectionsRequest` filter builder (`.for_topic`/`.for_topics`/`.name_contains`/`.id_prefix`, then `.list().await` -> `Vec<ProjectionInfo>`, no filter lists all) over the `AGDX_GET_PROJECTION`/`AGDX_LIST_PROJECTIONS` managed commands (query block, 1_000_110/111). `ProjectionInfo` is the `Projection` (schema, content type, indexed fields) plus the bindings routing into it. Filters are AND-composed by LaserData Cloud against its registry. LaserData-Cloud-only (gated on `capabilities.managed`), off the log via the managed command channel. CRUD is a write on the control topic, browse is a managed read. Writer schemas: `laser.schemas().register(source).send()` is SYNCHRONOUS over `AGDX_REGISTER_SCHEMA` (1_000_122). LaserData Cloud validates, allocates a collision-free id, durably appends, and returns the id. `laser.schemas().drop(id)` is an async control write (drop tombstones, ids permanent). `laser.schemas().get(id)` / `.list()` browse over `AGDX_GET_SCHEMA`/`AGDX_LIST_SCHEMAS` (1_000_120/121), returning `SchemaInfo { schema, dropped }`. `SchemaSource` is avro, protobuf, or json_schema.
- **HTTP API (future)**: projections CRUD plus browse, query, and KV are also exposed over REST on LaserData Cloud's HTTP API for browser/UI clients and a future public API. Thin translation onto the same managed ops, not a second source of truth. Contract: the AGDX spec.
- **Aggregate result keys**: each `AggCall`'s value lands under `Row.headers` keyed by its `alias` (the fluent `count()`/`sum(f)`/... default the alias to the function name). Group-by keys ride their own `Row.headers` entries, and a windowed aggregate adds a `window_start` header (the bucket lower edge, epoch micros).
- **Query index names must be `[A-Za-z0-9_]`**, since LaserData Cloud materializes each index under that exact name, so a dotted topic (`api.calls`) is rejected. Use `api_calls`.
- **Materialized source topics may have ANY number of partitions.** LaserData Cloud projects every partition of a source, so balanced or keyed publishing across N partitions all materializes.

## The indexing contract

- **`agdx.idx.<field>` headers are explicit index keys**, small scalars stamped by the producer (`Record::index`). The query DSL `field` names the part **after** `agdx.idx.`. Required only when no projection is bound to the topic. With a projection, the projector extracts fields from the payload.
- **`Projection` plus `ProjectionBinding`** is the operator-shape model:
  - `Projection::builder(id).name(...).version(n).content_type(ct).fields([..]).build()` builds the global reusable definition. `inline_payload` is ON by default, opt out with `.index_only()` (no body copy alongside the row). `.field(name)` / `.field_at(name, pointer)` are the single-field forms. `.fields(["a","b"])` is the bulk form.
  - `ProjectionBinding::builder().source(stream, t).allow(ref).default_projection(ref).target_table(t).build()` binds it to a source `(stream, topic)` (both required, a topic only exists within a stream) and names the materialization target. `.selector(s)` takes a pre-built `SourceSelector`. `.try_build()` returns `Result` instead of panicking when the source is missing.
  - `.retention(RetentionPolicy)` decouples row lifetime from the source topic's Iggy `message_expiry`: `MirrorLog` (default, follow the log plus wipe on topic delete), `Keep` (forever), `KeepUntilSourceDeleted` (forever but wiped on topic delete), `TimeToLive { ttl_micros }`, `MaxRows { rows }`. Field is `Option<RetentionPolicy>` (`skip_serializing_if`, back-compat), unset inherits the managed default. Enforced managed-side. See the AGDX spec.
- **`agdx.ref`** (projection ref) is the per-record selector, an opaque string like `"order.v1"`. **Distinct from `agdx.sid`**: the schema id resolves the registered writer schema for codec dispatch (Avro/Protobuf decode in LaserData Cloud), projection_ref is platform-controlled materialization routing. Both can coexist on the wire.
- **Codec abstraction**: `Codec<T>` (encode) plus `Decoder<T>` (decode) traits with `Json` plus `Msgpack` built-ins. Encode is `?Sized` (you can encode a `&str`), decode is a separate trait because it must produce an owned, deserializable value. Custom codecs (Avro / Protobuf / CBOR / BSON / Arrow / framing) implement the half they need and run through `.encode_with::<C, _>(&body)?` (single) / `.add_encoded::<C, _>(&body)?` (batch) on the write side, and `fetch_typed_with::<C, _>` / `Row::decode_with::<C, _>` on the read side. Raw bytes path: `.raw_bytes(bytes, ContentType::*)`. Byte inputs are `impl AsRef<[u8]>` / `impl Into<Vec<u8>>` and `Row.payload` is `Option<Vec<u8>>`, the public API never exposes the `bytes` crate.
- **`agdx.inline`** controls whether the body rides alongside the row in the index DB. Default for projections built through `Projection::builder` is ON (so `fetch_typed::<T>()` works without a log replay), opt out per projection with `.index_only()`. Per-record overrides exist on `Record` for the header-driven escape hatch path.
- **Conventions**: `agdx.idx.message_type` backs `Query::message_type` (`FIELD_MESSAGE_TYPE`). `agdx.idx.ts` (epoch micros) backs `time_range` (`FIELD_TS`). `agdx.ct` / `agdx.sid` / `agdx.ref` are framing / routing headers, not index keys, they ride `Row::metadata`.
- A field that lives **only in the payload is not matchable** unless a projection declares its JSON pointer.

## Running

`just ci` runs the whole gate (fmt + sort + machete + clippy + workspace tests + integration). To run one piece: `just lint` / `just test` / `just test-it`. The query client surface is covered by unit tests over the DSL, the wire round-trip, and the `agdx.idx.*` lowering. Query DSL **semantics** (filter, order, pagination, aggregate) are covered by a pure in-memory reference engine (`laser_bdd::query_engine` (in the `bdd/` harness), no Iggy, no transport, no qr) through its own unit tests and the `bdd/scenarios/query.feature` scenarios, which is the cross-SDK contract for what a query must return. The full **managed execution** (the product engine, the projector) is verified in LaserData Cloud's own repository, not here, since that engine is managed-side and there is no offline query transport.
