# Key-value store (client)

The TypeScript peer is `foreign/typescript/src/managed/kv.ts` over
`src/wire/kv.ts`, including CAS, fenced CAS, scans, expiry, patch, and leases.

A small managed key-value store next to the query layer, reachable on the one Iggy connection. Like `query`, the SDK ships only the **client**. The store lives in LaserData Cloud's managed point-state backend and the Iggy server forwards the ops. All of it is behind the independently selectable `kv` feature: private managed-command plumbing is shared without a semantic feature dependency. Wire: the AGDX spec

Authorization: the KV ops are gated by the fork-native capability layer (`kv:read` for get/scan/exists/namespaces, `kv:write` for set/cas/patch/expire/lease/copy/move, `kv:delete` for delete/delete-many), scoped by the **namespace** as the keyed resource. The streaming server derives `(feature, action)` from the command code and the namespace from the request and checks the caller's grants before forwarding (see the wire `authz` module and the AGDX spec's namespace-scoped grant rules). A **memory read** rides this as `kv:read` on the materialized memory namespace. A memory **write** is a streaming publish on the memory topic (gated by Iggy `send_messages`), not a KV capability.

## What ships here (`sdk/src/kv/mod.rs`, the fluent builders in `sdk/src/kv/client.rs`)

- `Laser::kv(namespace)` -> a cheap namespace-scoped `Kv` handle owning a `Laser` clone (`'static`, so it moves into a spawn or stores in a durable `StateStore` seam). `Laser::kv_namespaces()` lists every namespace holding at least one entry for the caller (read-only, user-scoped, for tooling/UI browsing).
- Reads: `get` (`Option<Vec<u8>>`), `get_entry` (`KvEntry` with expiry), `get_typed::<T>` (JSON-decode), `get_as::<C, T>` (decode with any `Decoder`), `exists(key)` (`Option<KvMetadata>`: version/expiry/size without transferring the value, the cheap precondition check before a large fetch).
- `set(key)` builder: `.bytes(impl AsRef<[u8]>)` / `.json(&v)?` / `.msgpack(&v)?` / `.encode_with::<C, _>(&v)?` (any `Codec` - Avro / Protobuf / Arrow / your own), optional `.ttl(Duration)` / `.expires_at(epoch_micros)`, then `.send()`. Values are opaque bytes, so the codec is the caller's choice on both ends.
- Compare-and-swap on the same `set` builder: `.expect_version(n)` (apply only if the key holds version `n`) or `.expect_absent()` (create-if-absent), then `.commit().await -> u64` (the new version). A precondition miss is `KvError::VersionConflict { current: Option<u64> }` (`LaserError::is_version_conflict()`) carrying the present version (`None` = absent), so the caller re-reads and retries. `KvEntry.version` (from `get_entry`) is the compare token, and `0` means an unversioned store.
- `cas_fenced(key, fence_key, fence_token)` builder: the same value/precondition/expiry chain as `set`, finished with `.commit().await -> u64`, but the write also applies only while `fence_key`'s task-fence sequence still equals `fence_token` (the token a `lease` granted). The at-most-one-effective-writer gate for an exclusive workflow step (see `workflow.rs` in [agent-runtime](../agent-runtime/SKILL.md)): a stale holder is rejected as `KvError::LeaseLost` even if its lease was presumed lost, a precondition miss is still `KvError::VersionConflict`.
- `expire(key, ttl)` sets or clears an entry's expiry in place without rewriting the value, returning its unchanged version. `patch(key, patch_bytes)` applies a codec-specific merge patch (a JSON merge patch over a JSON value, for instance) without transferring the whole object, returning the new version.
- `lease(key, ttl) -> Lease { token, granted_ttl }` acquires an advisory bounded-TTL lock, the fencing token to present to `cas_fenced`. `release(key, token) -> bool` releases it early.
- `delete(key)` -> `bool` (existed). `delete_many()` builder (`.prefix` / `.range` / `.key_contains`, same bounds as scan) -> count removed. No bounds clears the namespace. `scan()` builder: `.prefix` / `.range` (byte order) / `.key_contains(str)` (valid-UTF-8 keys only) / `.limit` / `.cursor` -> `.fetch()` (one `KvPage`) or `.entries()` (walk all pages by cursor). Both `scan()` and `delete_many()` also take `.conversation(id)` (the conversation lens): narrow to the rows one conversation wrote over a memory-view namespace, a filter on the auto-projected `conversation_id` field. A generic entry carries no conversation and is left out of a conversation-filtered scan.
- `copy_to(key, to_key)` / `move_to(key, to_key)` builders (`.into_namespace(ns)` for a cross-namespace target, then `.send() -> u64` the destination's new version): one backend transaction under the store's single-writer guard, move is copy plus source delete, absent-or-expired source is `KvError::NotFound`, destination overwritten (guarded copy composes `exists` + `cas`), the value moves with its remaining expiry. `get_many(keys) -> Vec<Option<Vec<u8>>>` rides the mixed-operation batch (`Laser::execute_batch`, `AGDX_BATCH_CODE` = base+20, up to `MAX_BATCH_OPS` = 64 items, per-op results, never a transaction, nested rejected). Python mirrors all three (`copy_to`/`move_to` with `to_namespace=`, `get_many`).
- Keys / values cross the API as `impl AsRef<[u8]>` in, `Vec<u8>` out (no `bytes` crate leak). `KvEntry { key: Vec<u8>, value: Vec<u8>, expires_at_micros, version, scope }`, `key_str()` gives the UTF-8 view when the key is valid UTF-8, `decode_value`/`decode_value_with::<C, _>` decode the value. `scope` is `Option<Box<MemoryRowScope>>`, present only on a memory read-view row and `None` on a generic entry: the memory provenance (kind, agent, user, app, conversation) plus a `SourceRef` source pointer to the origin log record, boxed so a generic entry stays small. Agentic memory recall reads these back off the shared KV read view.
- `execute_kv(namespace, code, request)` (crate-internal) gates on `capabilities.kv.available` (set by the connect-time `AGDX_HELLO` probe against a managed deployment), encodes the request as CBOR, sends it as the op's managed command via `Laser::send_raw_with_response`, decodes `KvReply`.

## Wire (the managed KV contract)

- One managed command **per op** in the KV block (base `AGDX_KV_BASE` = 1_000_300): `AGDX_KV_GET_CODE` (+0), `AGDX_KV_SET_CODE` (+1), `AGDX_KV_SCAN_CODE` (+2), `AGDX_KV_DELETE_CODE` (+3), `AGDX_KV_DELETE_MANY_CODE` (+4), `AGDX_KV_NAMESPACES_CODE` (+5), `AGDX_KV_CAS_CODE` (+6), `AGDX_KV_EXISTS_CODE` (+7), `AGDX_KV_EXPIRE_CODE` (+8), `AGDX_KV_PATCH_CODE` (+9), `AGDX_KV_LEASE_CODE` (+10), `AGDX_KV_RELEASE_CODE` (+11), `AGDX_KV_CAS_FENCED_CODE` (+12), `AGDX_KV_COPY_CODE` (+13), `AGDX_KV_MOVE_CODE` (+14). `KV_OP_VERSION` = 1 (every op past the base seven is additive: an incapable backend rejects the code -> `Unsupported`, advertised by its own capability flag, not a version bump).
- Requests: `KvGet` / `KvSet` / `KvCas` / `KvCasFenced` / `KvDelete` / `KvScan` / `KvDeleteMany` / `KvExists` / `KvExpire` / `KvPatch` / `KvLease` / `KvRelease` / `KvCopy` (CBOR-named, each carries `v` + `namespace`). `KvScan`/`KvDeleteMany` carry an optional `conversation` (the conversation lens, additive and omitted on the wire when unset). `KvCas` adds `expect: CasExpect = Match(u64) | Absent`, and `KvCasFenced` adds `fence_key` + `fence_token` on top of that. Reply: `KvReply = Ok(KvOutcome) | Err(KvError)` where `KvOutcome = Value(Option<KvEntry>) | Written | Committed { version } | Versioned { version } | Deleted(bool) | DeletedMany(usize) | Page(KvPage) | Namespaces(..) | Metadata(Option<KvMetadata>) | Leased { lease_token, granted_ttl_micros } | Released(bool)` and `KvError` adds `VersionConflict { current: Option<u64> }` and `LeaseLost` (a fenced write whose token a newer holder superseded). `KvEntry` carries `version: u64` (skip-when-zero).
- `key` and `value` are arbitrary bytes, a CBOR byte string (byte-exact). Scan bounds (`prefix`/`start`/`end`/`cursor`) are bytes, `key_contains` is a string.
- The managed deployment wraps each op in the shared `ForwardedCommand { user_id, client_id, correlation, read_all, command_code, payload }` keyed frame (also used by registry browse, and `AGDX_QUERY` keeps its own `ForwardedQuery`), stamping identity. The forwarded-frame shapes live in laser-wire (`wire/src/forward.rs`). The dispatch and the backend are managed-side behavior, not in this repo.

## Conventions + caps

- **Keys** arbitrary bytes, max 512 B. **Values** arbitrary opaque bytes, max 8 MiB (bounded so the embedded store never bloats). Scan limit 1000 (default 100). The SDK validates key length + value size before sending. Oversized fails fast as `LaserError::Kv`. Byte-prefix/range scans always work. `key_contains` needs LaserData Cloud to index a UTF-8 form of the key, so it matches only keys that are valid UTF-8.
- A **namespace** is a logical bucket (keys unique within it, scans scoped to it, isolated by LaserData Cloud). Expiry is lazy-on-read + swept, managed-side.
- KV is LaserData Cloud only. Raw Apache Iggy (no managed backend) -> `capabilities.kv.available` false -> `LaserError::Unsupported`.
- **CAS is backend-gated.** Where the backend supports conditional writes, CAS is served as a single conditional update. A backend that cannot do a conditional write leaves the `kv_cas` capability clear. The flag is advertised on the binary `AGDX_HELLO` reply (`OpVersions.features` bit `feature::KV_CAS`) and surfaced as `laser.capabilities().await.kv.cas`. `commit()` pre-gates on it and returns `LaserError::Unsupported` when it is clear, so a deployment that serves CAS MUST advertise the bit. CAS also rides its own command code, so even an unaware server rejects it cleanly. Fenced CAS is the same pattern one level up: `feature::KV_CAS_FENCED` / `capabilities.kv.cas_fenced`, checked by `cas_fenced(..).commit()`.

## Rules specific to this area

- KV is mutable **point state**, NOT a message log: it does not ride the Iggy log. Do not route set/delete through topics. (Contrast `query`, whose writes ARE log appends.) LaserData Cloud picks the durability and multi-node options.
- The SDK is **client-only**: never add a KV backend/worker here. Storage, expiry, and dispatch are server-side concerns.
- Keep boundaries crisp: log = history/stream, KV = point state, AgentFS (future) = per-agent filesystem/workspace. Do not conflate.
- KV is a **primitive**, not the agent "memory" abstraction (which is the `Memory` trait in `sdk/src/memory.rs`). The two are distinct: memory (including its named-item altitude `MemoryHandle::{set,fetch,update,remove}`) always rides the memory topic as durable records, which a deployment materializes into the versioned KV read view. `Laser::kv` is the raw primitive, a direct point store with no stream. Add memory semantics behind the `Memory` trait - never rename `kv` to `memory`.
- Tests live in-module (`sdk/src/kv/client.rs` `#[cfg(test)]`): codes, envelope round-trips, key validation, error mapping. BDD names, `.expect("msg")`.
